Chapter 1 · Slides 1–17

Introduction to Database Systems

What is a DBMS, why use one, architecture, data independence

What Is a DBMS? Slides 2–4

A Database Management System (DBMS) is a software system designed to store, manage, and retrieve very large amounts of data efficiently and safely. Think of it as a super-powered filing cabinet that millions of people can use at the same time without stepping on each other's toes — and that can recover from a power outage without losing any data.

A database models real-world enterprises: it stores entities (things that exist — like students, courses, employees) and relationships between them (e.g., "Madonna is enrolled in CS281").

Why not just use files (like Excel or text files)? Files give you nothing for free. You'd have to write your own code for: concurrent access, crash recovery, security, efficient search, and keeping data consistent. The DBMS handles all of this.

Examples
Example 1 — University database

Bilkent's student system is a database. It stores Students, Courses, Enrollments, Professors, Grades. When thousands of students log in to see their grades at the same time, the DBMS handles all those simultaneous requests without returning wrong data to anyone.

Example 2 — Banking system

When you use an ATM, a bank's DBMS handles your transaction. If two people simultaneously withdraw from the same account, the DBMS ensures neither transaction interferes with the other — you can't both withdraw more money than exists. If the ATM crashes mid-transaction, the DBMS rolls back as if it never happened.

Example 3 — File system problems

Imagine storing student grades in a text file. Problem 1: If two professors update grades at the same moment, one update gets lost. Problem 2: If the computer crashes while writing, the file is half-written and corrupted. Problem 3: To find all students with GPA > 3.5, your code must read every line. A DBMS solves all three with no custom code from you.

Levels of Abstraction Slides 7–9

A DBMS provides three levels of abstraction, allowing different people to see the database differently without affecting each other:

LevelWhat It IsWho Uses ItExample
Physical Schema How data is actually stored on disk — which files, which indexes, which pages. DBA (database administrator) "Students stored as unsorted heap file; B+ tree index on sid."
Conceptual Schema The logical structure — what tables exist, what columns they have, what constraints hold. Developers, DBAs Students(sid, name, login, age, gpa), Enrolled(sid, cid, grade)
External Schema (View) A customized "window" into the database for specific users — shows only what they need. End users, applications A student sees only their own grades. A professor sees the class roster.
🔑
Data Independence

Physical independence: Change how data is stored on disk (e.g., add an index, switch disk format) without changing any application code. Logical independence: Add a new column to a table without breaking applications that don't use that column. This is one of the biggest wins of using a DBMS.

Examples
Example 1 — Physical independence

The database administrator decides to add a B+ tree index to the Students table to make searches faster. Zero lines of application code change — the application still says "SELECT * FROM Students WHERE gpa > 3.5" and gets the same result, just faster. The app doesn't know or care how data is physically stored.

Example 2 — Logical independence

The DBA adds a new column "phone_number" to the Students table. Existing applications that only query sid, name, and gpa continue working without any modification — they just get back the same columns they always did.

Example 3 — Views for security

HR database: The full Employees table has salary, SSN, performance rating. A manager's view only shows employee names and departments. A payroll clerk's view shows only name and salary. Neither can see the other's data. Each "sees" a different, safe subset of the same underlying table.

DBMS Architecture — Layered System Slide 16

A DBMS is built in layers, like an onion. Each layer uses the services of the one below it and provides services to the one above:

TOP: User/Application ↕ Query Optimization & Execution ← Finds the fastest way to run a query ↕ Relational Operators ← Executes joins, selects, projects ↕ Files & Access Methods ← Reads/writes pages from/to disk ↕ Buffer Management ← Caches disk pages in RAM ↕ Disk Space Management ← Allocates space on the physical disk ↕ BOTTOM: Physical Disk
Examples
Example 1 — A SELECT query's journey

You type: SELECT name FROM Students WHERE gpa > 3.5. The query optimizer (top layer) decides whether to use an index or scan the file. It tells the relational operator layer to execute a "selection." That layer asks the file/access layer for the right pages. The buffer manager checks if those pages are already in RAM. If not, disk space management fetches them. Data bubbles back up to you.

Example 2 — Buffer management analogy

The buffer manager is like your desk while working in a library. You don't want to walk to the shelves every time you need a page — instead you keep frequently-needed books on your desk (in RAM). The buffer manager decides which pages to keep "on the desk" and which to send back to the shelves (disk).

Example 3 — Concurrency and recovery are everywhere

Notice the slide says concurrency control and recovery span ALL layers. When you read a page (buffer layer), you may need a lock. When you write a page (disk layer), you must write a log record first. These concerns don't live in just one layer — they cut across the entire system.

📋 Chapter 1 Summary

  • A DBMS stores large datasets and handles concurrency, crash recovery, security, and efficient querying — so you don't have to.
  • Three levels: Physical (how stored), Conceptual (what exists), External (what users see).
  • Data independence = change physical/logical structure without breaking applications.
  • A DBMS has a layered architecture: disk → buffer → files → operators → optimizer.
  • Concurrency control and logging run through all layers.
Chapter 2 · Slides 18–35

The Entity-Relationship Model

Entities, Relationships, Keys, ISA, Weak Entities, Aggregation, ER-to-Relational

ER Model Basics — Entities and Relationships Slides 19–22

The ER Model is a visual way to design a database before writing any code. You sketch out the "things" that exist in your system and how they relate to each other. It's the blueprint before you build the house.

  • Entity: A real-world "thing" that is distinguishable from other things. Examples: a student, a course, an employee, a book.
  • Entity Set: A collection of similar entities. Like a class of all students. All entities in a set share the same attributes.
  • Attribute: A property of an entity. For a student: name, age, GPA, student ID. Each attribute has a domain (allowed values).
  • Key: One or more attributes that uniquely identify an entity within its entity set. For students, the student ID (ssn/sid) is a key.
  • Relationship: An association between entities. "Attishoo works in the Pharmacy department."
  • Relationship Set: A collection of similar relationships. "Works_In" is a relationship set connecting all (employee, department) pairs.
📐
ER Diagram Notation (know this!)

Rectangles = Entity sets. Ovals = Attributes. Diamonds = Relationship sets. Lines connect them. Underlined attributes = key attributes. Double rectangles/diamonds = weak entities and identifying relationships.

Examples
Example 1 — University ER

Entity sets: Students (attributes: sid, name, login, age, gpa) and Courses (cid, cname, credits). Relationship: Enrolled (which students are in which courses, with descriptive attribute: grade). The "grade" lives on the relationship, not on either entity — it only makes sense in the context of a specific student AND a specific course.

Example 2 — Employee company ER

Entity sets: Employees (ssn, name, lot) and Departments (did, dname, budget). Relationships: Works_In (employee works in department, descriptive attribute: since) and Manages (one manager per department, descriptive attribute: since). Same entity sets, different relationships!

Example 3 — Roles in a relationship

The Reports_To relationship connects Employees to Employees — same entity set appears twice! One instance plays the "supervisor" role, the other plays the "subordinate" role. Roles let one entity set participate multiple times in a relationship.

Key Constraints and Participation Constraints Slides 23–25

Constraints define how many times an entity can participate in a relationship.

  • Key constraint (arrow in ER): An entity set participates in a relationship at most once. "Each department has at most one manager." Arrow points FROM the "one" side.
  • Participation constraint (thick line): Every entity in the set must participate in the relationship. "Every department MUST have a manager." Thick line = total participation.
Relationship TypeMeaningER Notation
One-to-One (1:1)Each entity in A maps to at most one in B, and vice versa.Arrow on both sides of diamond
One-to-Many (1:N)One entity in A can relate to many in B, but each B relates to at most one A.Arrow on the "one" side
Many-to-Many (M:N)Entities on both sides can relate to many on the other side.No arrows
Examples
Example 1 — Works_In is Many-to-Many

An employee can work in MANY departments (e.g., half-time in CS, half-time in EE). A department can have MANY employees. No arrows on either side of Works_In. Both entity sets have partial participation (not every employee must work somewhere, not every department must have employees — though in practice they do).

Example 2 — Manages is Many-to-One with total participation

Each department has AT MOST ONE manager (key constraint: arrow from Departments to Manages). Also, EVERY department must have a manager (total participation: thick line from Departments to Manages). But a manager can manage at most one department... or can they? If no arrow from Employees, a manager could manage multiple departments.

Example 3 — Marriage analogy

Consider a "Married_To" relationship between People. In a monogamous society: One-to-One (each person married to at most one other). Each marriage involves exactly two people. Participation is partial: not everyone is married. If we said "every adult must be married," participation would be total (thick line).

Weak Entities Slide 26

A weak entity is an entity that cannot be uniquely identified by its own attributes alone — it depends on a related "owner" entity. Think of dependents (children/spouse) on an employee's insurance policy. A dependent named "John" isn't unique — there could be thousands of Johns. But "John, dependent of employee 12345" is unique.

  • The weak entity's partial key (called a discriminator) uniquely identifies it among the owner's dependents, but not globally.
  • The identifying relationship is always one-to-many (one owner, many weak entities).
  • The weak entity set has total participation in the identifying relationship — it can't exist without its owner.
Examples
Example 1 — Dependents on insurance (the slides' example)

Entity: Employees (key: ssn). Weak Entity: Dependents (discriminator: pname = dependent name). A dependent is identified by (ssn of owner, pname). If employee 123-45-6789 is deleted, all their dependents are automatically deleted too (CASCADE). The dependent literally cannot exist without the employee.

Example 2 — Order items

An Order contains multiple OrderItems. An item (Product ID + quantity) only makes sense in the context of a specific order. OrderItem is a weak entity with the Order as its owner. The item's "line number" (1, 2, 3...) is the discriminator — unique within one order, but not globally.

Example 3 — Apartment in building

An Apartment has a unit number (101, 202, 305). But unit 101 exists in EVERY building — it's not globally unique. The weak entity Apartment gets its identity from (Building_ID, unit_number). Delete the building, all apartments go with it.

ISA ("Is A") Hierarchies Slides 27–28

ISA hierarchies work exactly like class inheritance in object-oriented programming (like C++ or Java). If A ISA B, every A is also a B, and A inherits all of B's attributes while having additional attributes of its own.

  • Overlap constraint: Can an entity be in multiple subclasses at once? "Can Joe be both an Hourly_Emp AND a Contract_Emp?" — the model specifies allowed or disallowed.
  • Covering constraint: Must every entity in the superclass be in at least one subclass? "Must every Employee be either Hourly or Contract?" — yes or no.
Examples
Example 1 — Employee hierarchy (the slides' example)

Superclass: Employees (ssn, name, lot). Subclass 1: Hourly_Emps (inherits ssn/name/lot, adds: hourly_wages, hours_worked). Subclass 2: Contract_Emps (inherits ssn/name/lot, adds: contractid). Every Hourly_Emp IS an Employee. But not every Employee needs to be Hourly or Contract (partial covering).

Example 2 — Animal taxonomy

Superclass: Animals (name, age, weight). Subclass: Dogs (adds: breed). Subclass: Cats (adds: indoor/outdoor). Every Dog IS an Animal and inherits all Animal attributes. Covering: Maybe not every animal is a dog or cat — covering is not total. Overlap: In this model, an animal can't be both a dog and a cat — disjoint.

Example 3 — Why use ISA?

Reason 1: To add attributes specific to a subtype (Hourly_Emps need hours_worked, Contract_Emps don't). Reason 2: To identify entities that participate in specific relationships (only Contract_Emps sign contracts; only Hourly_Emps punch time cards). Without ISA, you'd store null values everywhere — messy and wasteful.

Aggregation Slide 29

Aggregation is used when you need to model a relationship that involves another relationship — treating a relationship set as if it were an entity. This is the trickiest ER concept.

Example from slides: Employees monitor the Sponsors relationship between Departments and Projects. You can't just put "Monitors" as a ternary relationship — because "Monitors" has its own attribute (until) and you want to say "each sponsorship is monitored by at most one employee."

Examples
Example 1 — Employee monitors a sponsorship

The relationship Sponsors links Departments and Projects. We want to say an Employee monitors a particular (Dept, Project) sponsorship — and track "until when" they monitor it. Solution: Aggregate Sponsors into a box, and draw Monitors as a relationship between Employees and that aggregated box. The sponsorship becomes a "thing" that can be related to.

Example 2 — Doctor reviews a prescription

Prescription is a relationship between Doctor and Patient. A PharmacyReviewer reviews specific prescriptions. To model "Reviewer reviews Prescription" — you need aggregation: treat the Prescription relationship as an entity, then have a Reviews relationship from PharmacyReviewer to (aggregated) Prescription.

Example 3 — When aggregation vs. ternary?

Use a ternary relationship when all three entities participate equally with no substructure. Use aggregation when one relationship participates in another relationship AND there's something special about that nested relationship (own attributes, key constraints on participation). Aggregation is richer — it lets you say "at most one employee monitors each sponsorship."

Design Choices — Entity vs. Attribute, Binary vs. Ternary Slides 30–34

ER design is subjective. There's often no single correct answer, but there are guidelines that avoid common mistakes.

Entity vs. Attribute:

  • If a concept has its own structure (multiple fields, or multiple values per entity) → model as an entity.
  • If it's a simple, single-valued property → model as an attribute.
Examples
Example 1 — Address: entity or attribute?

If each employee has exactly one address and you only need it as a string → attribute. If employees can have multiple addresses, or you need to query by city → model Address as an entity (with street, city, country fields) connected by a "LivesAt" relationship. Attributes can't be multi-valued or have sub-structure.

Example 2 — Works_In with time periods

If you want to record that an employee worked in a department for multiple periods (Jan–Mar, Sept–Dec), the "from/to" dates can't just be attributes on Works_In — they'd need to repeat. Solution: introduce a Duration entity set and make Works_In4 a relationship connecting Employees, Departments, and Duration. Or redesign the relationship itself.

Example 3 — Binary vs. Ternary: Insurance policies

Bad design: single ternary relationship Covers(Employee, Policy, Dependent). This says each policy covers each combination of employee and dependent. Better: Two binary relationships — Purchaser(Employee, Policy) and Beneficiary(Dependent, Policy). Why? If each policy is owned by exactly one employee (key constraint), the binary design captures this naturally; the ternary doesn't.

Translating ER Diagrams to Relational Tables Slides 35–49

Once you have an ER diagram, you must convert it to actual relational tables (SQL CREATE TABLE statements). Here are the rules:

ER ConstructTranslation Rule
Entity setOne table. Columns = attributes. Primary key = the key attribute(s).
Many-to-Many relationshipOne new table. Columns = keys of each participating entity set (as foreign keys) + descriptive attributes. Primary key = combination of both foreign keys.
Key constraint relationship (1:N)Option 1: Relationship table where the "one" side's key is the primary key. Option 2: Merge the relationship into the "one" entity's table (usually better).
Total participationAdd NOT NULL constraint on the foreign key column.
Weak entity setMerge with identifying relationship into one table. Primary key = owner's key + discriminator. Add ON DELETE CASCADE.
ISA hierarchyOption 1: Superclass table + subclass tables (each with ssn FK + own attrs). Option 2: Just subclass tables with all inherited attrs. Trade-off: Option 1 is better for queries over all employees; Option 2 avoids joins for subclass queries.
Examples
Example 1 — Many-to-many: Works_In
CREATE TABLE Works_In ( ssn CHAR(11), did INTEGER, since DATE, PRIMARY KEY (ssn, did), -- combination is the key FOREIGN KEY (ssn) REFERENCES Employees, FOREIGN KEY (did) REFERENCES Departments );
Example 2 — Key constraint: Manages (merged)

Since each department has at most one manager, we can merge Manages into Departments:

CREATE TABLE Dept_Mgr ( did INTEGER PRIMARY KEY, dname CHAR(20), budget REAL, ssn CHAR(11) NOT NULL, -- total participation: every dept HAS a manager since DATE, FOREIGN KEY (ssn) REFERENCES Employees ON DELETE NO ACTION );
Example 3 — Weak entity: Dependents
CREATE TABLE Dep_Policy ( pname CHAR(20), age INTEGER, cost REAL, ssn CHAR(11) NOT NULL, PRIMARY KEY (pname, ssn), -- discriminator + owner key FOREIGN KEY (ssn) REFERENCES Employees ON DELETE CASCADE -- deleting an employee deletes all their dependents );

The Relational Model

Relations, Keys, Integrity Constraints, Foreign Keys, SQL DDL, Views

Relations — Tables with Rows and Columns Slides 37–40

In the relational model, everything is a table. A relation has two parts:

  • Schema: The structure — name of the table and name + type of each column. Like a blueprint. Written as: Students(sid: string, name: string, login: string, age: integer, gpa: real)
  • Instance: The actual data — the set of rows (tuples) currently in the table. Changes over time as data is inserted/deleted/updated.

Important terms: cardinality = number of rows. degree (arity) = number of columns. A relation is a set of tuples — all rows are distinct (no exact duplicates).

Examples
Example 1 — Students table

Schema: Students(sid: string, name: string, login: string, age: int, gpa: real). Instance (3 rows, degree 5):

sid | name | login | age | gpa ------+-------+------------+-----+----- 53666 | Jones | jones@cs | 18 | 3.4 53688 | Smith | smith@eecs | 18 | 3.2 53650 | Smith | smith@math | 19 | 3.8

Note: Two "Smiths" — that's fine, name is not a key. But no two rows can have the same sid.

Example 2 — No ordered rows

A relation is a mathematical SET — rows have no inherent order. There is no "first row" or "last row" in the relational model. When you query, the database may return rows in any order unless you specify ORDER BY. This is important: don't rely on insertion order!

Example 3 — All columns must be atomic

In the pure relational model, every cell contains a single atomic value — no lists, no nested tables, no arrays. You can't store "Smith's courses: {CS281, CS315}" in one cell. Instead, you'd have a separate Enrolled table with one row per (student, course) pair. This is called First Normal Form (1NF) — the most basic requirement of the relational model.

Keys, Superkeys, and Candidate Keys Slides 41–43

A key is the most important concept in database design. It's a set of attributes that uniquely identifies every row in a table.

  • Superkey: Any set of attributes that uniquely identifies rows. {sid, name} is a superkey (no two rows have same sid AND name). But it's redundant — sid alone already does the job.
  • Candidate key: A MINIMAL superkey. No proper subset of it is also a superkey. {sid} is a candidate key; {sid, name} is not (it's a superkey but not minimal).
  • Primary key: One chosen candidate key, declared in the schema. The DBA picks one if multiple candidate keys exist.
Examples
Example 1 — Identifying keys in Students

Students(sid, name, login, age, gpa). Candidate keys: {sid} and possibly {login} (if logins are unique). NOT a key: {name} — two students can have the same name. NOT minimal: {sid, gpa} — gpa is redundant since sid alone identifies the row. Primary key chosen: sid.

Example 2 — Composite primary key

Enrolled(sid, cid, grade). Neither sid alone nor cid alone is a key — many students can enroll in the same course, and one student can enroll in many courses. The primary key is {sid, cid}: the combination. This means: "for a given student in a given course, there's one grade." That's the intended business rule.

Example 3 — Keys enforce real-world rules

If you declare sid as the primary key of Students, the DBMS will REJECT any attempt to insert two rows with the same sid. This is automatic enforcement of the real-world rule "each student has a unique ID." Without this, your application code would have to check manually — error-prone and tedious.

Foreign Keys and Referential Integrity Slides 44–48

A foreign key is a set of columns in one table that refer to the primary key of another table. It's a "logical pointer" — it says "this row relates to that row in another table." Referential integrity means no dangling references: every foreign key value must match an existing primary key value in the referenced table.

What happens when you delete a referenced row? Four options:

  • NO ACTION (default): Reject the delete — refuse to delete if any other row references it.
  • CASCADE: Automatically delete all rows that reference the deleted row.
  • SET NULL: Set the foreign key columns to NULL in referencing rows.
  • SET DEFAULT: Set the foreign key columns to their default value.
Examples
Example 1 — Student enrollment foreign key
CREATE TABLE Enrolled ( sid CHAR(20), cid CHAR(20), grade CHAR(2), PRIMARY KEY (sid, cid), FOREIGN KEY (sid) REFERENCES Students );

Now if you try to INSERT a row into Enrolled with sid=99999 and no student has sid=99999, the DBMS rejects it. Referential integrity is automatically enforced!

Example 2 — CASCADE on delete

Policy: if a student leaves the university, delete all their enrollment records too. Use ON DELETE CASCADE. When you DELETE FROM Students WHERE sid=53666, all rows in Enrolled with sid=53666 are automatically deleted. No orphaned enrollment records.

Example 3 — HTML links have no referential integrity!

The slides point out: web links in HTML have no referential integrity. You can have a hyperlink pointing to a page that doesn't exist (a 404 error). Relational databases enforce that every "link" (foreign key) points to something that actually exists. This is why databases are reliable and the web is... not always.

Views — Virtual Tables Slides 50–51

A view is a named query stored in the database. It looks and behaves like a table, but it stores no data itself — it computes its "rows" on the fly by running the underlying query. Views are used for security and convenience.

Examples
Example 1 — Creating a view
CREATE VIEW YoungActiveStudents (name, grade) AS SELECT S.name, E.grade FROM Students S, Enrolled E WHERE S.sid = E.sid AND S.age < 21;

Now you can query YoungActiveStudents just like a table. The DBMS runs the underlying SELECT behind the scenes. No data is stored — it's computed fresh each time.

Example 2 — Views for security

Employees table has: name, salary, SSN, performance_review. You create a view EmployeeDirectory with only: name, department, phone. You give external staff access to EmployeeDirectory but NOT to Employees. They can look up contact info without seeing salaries or SSNs. Views are a powerful security mechanism.

Example 3 — Views and DROP TABLE

If you DROP TABLE Students, what happens to views that use it? The database offers options: RESTRICT (refuse to drop if views depend on it) or CASCADE (drop the table AND all dependent views). Dropping views: DROP VIEW YoungActiveStudents — this only drops the view definition, NOT the underlying data in Students or Enrolled.

Chapter 4 · Slides 69–100

Relational Algebra

Selection, Projection, Cross-Product, Joins, Union, Intersection, Difference, Division

What Is Relational Algebra? Slides 70–72

Relational Algebra is the mathematical language for querying relational databases. It's operational — it tells the computer how to compute a result step by step. SQL is based on relational algebra under the hood; the query optimizer converts SQL into algebraic expressions to find the best execution plan.

Every operator takes one or more relations as input and produces a new relation as output. Because output is always a relation, you can compose operations: feed the output of one into the input of another. The algebra is "closed."

📖
Reference relations used throughout the examples

Sailors S1: (22, dustin, 7, 45.0), (31, lubber, 8, 55.5), (58, rusty, 10, 35.0)
Sailors S2: (28, yuppy, 9, 35.0), (31, lubber, 8, 55.5), (44, guppy, 5, 35.0), (58, rusty, 10, 35.0)
Reserves R1: (22, 101, 10/10/96), (58, 103, 11/12/96) — format: (sid, bid, day)

Selection (σ) — Filtering Rows Slide 76

Selection filters rows based on a condition. Written as σcondition(R). It keeps only the rows where the condition is true. The output relation has the same columns as the input but fewer rows.

Examples
Example 1 — Select sailors with rating ≥ 8

σrating ≥ 8(S2) returns rows from S2 where rating ≥ 8. Result: (28, yuppy, 9, 35.0) and (58, rusty, 10, 35.0). Rows with rating 5 and 8 (lubber, rating=8: keep! guppy, rating=5: remove!).

Example 2 — Compound condition

σrating > 8 AND age < 40(S2). Must satisfy BOTH conditions. (28, yuppy, 9, 35.0): rating=9>8 ✓, age=35<40 ✓ → included. (58, rusty, 10, 35.0): rating=10>8 ✓, age=35<40 ✓ → included. Yuppy and rusty make the cut.

Example 3 — SQL equivalent

σrating ≥ 8(S2) is equivalent to: SELECT * FROM S2 WHERE rating >= 8. The SQL WHERE clause IS the relational algebra selection operator. Every time you write a WHERE clause, you're applying a selection.

Projection (π) — Choosing Columns Slide 75

Projection keeps only specified columns (attributes). Written as πcol1, col2, ...(R). Output has the same rows but fewer columns. Because relations are sets, duplicate rows are eliminated.

Examples
Example 1 — Project name and rating

πsname, rating(S2): Keep only the sname and rating columns from S2. Result: (yuppy, 9), (lubber, 8), (guppy, 5), (rusty, 10). Four rows, two columns.

Example 2 — Duplicates eliminated

πage(S2): Keep only the age column. S2 has ages: 35.0, 55.5, 35.0, 35.0. After projection and duplicate elimination: {35.0, 55.5}. Two rows, not four! Relations are sets — no duplicates allowed.

Example 3 — SQL equivalent

πsname, rating(S2) ≡ SELECT DISTINCT sname, rating FROM S2. The SELECT clause IS the projection operator. DISTINCT is needed to eliminate duplicates (SQL by default keeps duplicates; relational algebra always eliminates them).

Cross-Product (×) and Joins Slides 78–80

The cross-product (Cartesian product) pairs every row of R with every row of S. If R has m rows and S has n rows, the result has m×n rows. This is rarely what you want alone — it's mostly used as a building block for joins.

A join is a cross-product followed by a selection. Joins are the most important operation in relational databases — they combine data from multiple tables.

  • Condition Join (θ-join): R ⋈c S = σc(R × S). Cross-product filtered by condition c.
  • Equi-Join: Condition join where c only uses equalities (=). One copy of equal-valued columns retained.
  • Natural Join: Equi-join on ALL common column names. Most common in practice.
Examples
Example 1 — Cross-product

S1 has 3 rows. R1 has 2 rows. S1 × R1 has 3 × 2 = 6 rows. Every sailor is paired with every reservation. Most pairings are nonsense (dustin is paired with rusty's reservation). The cross-product needs a join condition to make sense.

Example 2 — Natural join: find sailors who reserved boats

Sailors ⋈ Reserves: joins on the common column "sid." Result: only rows where a sailor's sid matches a reservation's sid. dustin (sid=22) joins with reservation (22, 101, 10/10/96). rusty (sid=58) joins with (58, 103, 11/12/96). lubber (sid=31) has no reservation → NOT in result.

Example 3 — SQL equivalent

S1 ⋈S1.sid=R1.sid R1 ≡ SELECT * FROM S1, R1 WHERE S1.sid = R1.sid. Every JOIN in SQL corresponds to a join in relational algebra. The DBMS optimizer decides the most efficient way to compute it (which table to scan first, whether to use indexes, etc.).

Union, Intersection, Set-Difference Slide 77

These set operations require union-compatible inputs: same number of columns AND corresponding columns have the same types.

  • Union (∪): All tuples that appear in R OR S (or both). No duplicates.
  • Intersection (∩): Only tuples that appear in BOTH R AND S.
  • Set-Difference (−): Tuples in R but NOT in S.
Examples
Example 1 — Union of two sailor tables

S1 ∪ S2: All sailors from either S1 or S2. S1 has dustin, lubber, rusty. S2 has yuppy, lubber, guppy, rusty. Union: dustin, lubber, rusty, yuppy, guppy — 5 unique sailors (lubber and rusty appear in both, but only once in result).

Example 2 — Set-difference

S1 − S2: Sailors in S1 but NOT in S2. S1 has sid 22 (dustin), 31 (lubber), 58 (rusty). S2 has 28, 31, 44, 58. Sailors in S1 not in S2 by sid: sid=22 (dustin). Result: just dustin.

Example 3 — SQL equivalents

Union → UNION in SQL. Intersection → INTERSECT. Set-difference → EXCEPT. Important: SQL UNION by default removes duplicates (like relational algebra). UNION ALL keeps duplicates. Most SQL systems support all three, though some older systems don't implement INTERSECT and EXCEPT directly.

Division (÷) — "For All" Queries Slides 81–84

Division answers queries of the form "find all X such that for every Y, (X, Y) is in the relation." It's used for "find all" / "reserved all" type queries. Not a primitive operator but very useful shorthand.

A ÷ B = all x values such that for every y in B, the pair (x, y) is in A.

Examples
Example 1 — Find sailors who reserved ALL boats

A = Reserves projected onto (sid, bid). B = Boats projected onto (bid). A ÷ B = sids of sailors such that for every boat bid in Boats, (sid, bid) is in Reserves. Only a sailor who reserved EVERY SINGLE BOAT appears in the result. Classic "for all" pattern.

Example 2 — Expressing division without the ÷ operator

Since division is just shorthand: A ÷ B = πx(A) − πx((πx(A) × B) − A). In words: take all x values. Remove x values that are "disqualified" by some y in B (i.e., where (x, y) is missing from A). What remains reserved all boats.

Example 3 — Division in SQL (using NOT EXISTS)
-- Find sailors who reserved ALL boats SELECT S.sname FROM Sailors S WHERE NOT EXISTS ( (SELECT B.bid FROM Boats B) -- all boats EXCEPT (SELECT R.bid FROM Reserves R -- boats this sailor reserved WHERE R.sid = S.sid) );

"Find sailors where there is NO boat that the sailor has NOT reserved." Double negative = "for all." This is the standard SQL pattern for division.

📋 Relational Algebra Operators Summary

  • σ (Selection): Filter rows. Like SQL WHERE.
  • π (Projection): Choose columns. Like SQL SELECT (with DISTINCT).
  • × (Cross-product): All row pairs from two tables. Used with selection to form joins.
  • ⋈ (Join): Cross-product + condition. Most common: equi-join and natural join.
  • ∪ ∩ − (Set ops): Union, intersection, difference. Need union-compatible schemas.
  • ÷ (Division): "For all" queries. Find X where every Y relates to X.
  • Algebra is closed: output is always a relation, so operators compose.
Chapter 5 · Slides 101–112

Relational Calculus

Domain Relational Calculus (DRC), Free/Bound Variables, Quantifiers, Relational Completeness

Algebra vs. Calculus — Operational vs. Declarative Slides 102–103

Relational Algebra: You specify how to compute the result step by step. Operational. "First do this, then do that."

Relational Calculus: You specify what you want, not how to compute it. Declarative. "Give me all X such that condition P(X) is true." This is exactly how SQL works — you describe what you want, the optimizer figures out how.

Two flavors: TRC (Tuple Relational Calculus — variables range over whole tuples) and DRC (Domain Relational Calculus — variables range over individual column values). The course focuses on DRC.

Examples
Example 1 — Algebra vs. calculus comparison

Same query, two approaches. Algebra: πsname(σrating>7(Sailors)) — "Apply selection, then project." Step-by-step. Calculus: {N | ∃I, T, A: ⟨I, N, T, A⟩ ∈ Sailors ∧ T > 7} — "Give me all names N such that there exists a sailor tuple with that name and rating > 7." Declare the answer, not the steps.

Example 2 — SQL is declarative calculus

SELECT sname FROM Sailors WHERE rating > 7 — this is declarative. You say what you want (sailors' names, filtering condition). The DBMS optimizer translates this into an efficient algebraic execution plan. You don't tell it to do a scan vs. use an index — it figures that out.

Example 3 — Power of declarativeness

In algebra, the order of operations matters for efficiency. In calculus, you say what's true and the optimizer picks the best order. This separation of "what" from "how" is one of the key innovations of the relational model — it makes query optimization possible and transparent to users.

DRC Formulas — Syntax and Quantifiers Slides 104–107

A DRC query has the form: {⟨x₁, x₂, ..., xₙ⟩ | p(x₁, x₂, ..., xₙ)}

"The set of tuples (x₁, ..., xₙ) such that formula p is true."

  • ∃X (p(X)): "There EXISTS some X for which p(X) is true." Used to say "there is some matching tuple somewhere."
  • ∀X (p(X)): "For ALL X, p(X) is true." Used for "for every" queries (like division).
  • Free variable: A variable that appears in the output (not bound by ∃ or ∀). These are what you're "looking for."
  • Bound variable: A variable bound by ∃ or ∀. Used to "check something" without being in the output.
Examples
Example 1 — Find sailors with rating > 7 (DRC)

{⟨I, N, T, A⟩ | ⟨I, N, T, A⟩ ∈ Sailors ∧ T > 7}

Variables I, N, T, A are all FREE — they appear on the left of | and must satisfy the formula. The formula says: this 4-tuple must come from Sailors, AND the rating T must be > 7. Result: all full sailor tuples where rating > 7.

Example 2 — Sailors rated > 7 who reserved boat 103

{⟨I, N, T, A⟩ | ⟨I, N, T, A⟩ ∈ Sailors ∧ T > 7 ∧ ∃Ir, Br, D (⟨Ir, Br, D⟩ ∈ Reserves ∧ Ir = I ∧ Br = 103)}

Now Ir, Br, D are BOUND by ∃ — we just check "does there exist a Reserves tuple matching this sailor and boat 103?" They don't appear in the output. I, N, T, A remain FREE (they are in the output).

Example 3 — Find sailors who reserved ALL boats (uses ∀)

{⟨I, N, T, A⟩ | ⟨I, N, T, A⟩ ∈ Sailors ∧ ∀B, BN, C (⟨B, BN, C⟩ ∈ Boats → ∃Ir, Br, D (⟨Ir, Br, D⟩ ∈ Reserves ∧ I = Ir ∧ Br = B))}

"A sailor is in the result if: for every boat B, IF B is a real boat (in Boats), THEN there exists a reservation by this sailor for B." The ∀ quantifier makes this a "for all" query — the DRC equivalent of division. B, BN, C, Ir, Br, D are all bound.

Unsafe Queries and Relational Completeness Slides 110–111

An unsafe query is a syntactically valid DRC query that has infinitely many answers. Example: {S | S ∉ Sailors} means "all tuples NOT in Sailors" — infinitely many! Safe queries restrict answers to values that actually appear in the database.

Relational Completeness: A query language is relationally complete if it can express every query that relational algebra can express. SQL is relationally complete — and actually more powerful (it has aggregates, which algebra doesn't).

Examples
Example 1 — Safe vs. unsafe

Unsafe: {⟨I⟩ | ¬(⟨I, N, T, A⟩ ∈ Sailors)} — "all IDs NOT belonging to any sailor." Could be infinitely many integers. Safe version: restrict to IDs that actually appear somewhere in the database domain.

Example 2 — RA and safe calculus are equivalent

Every query expressible in relational algebra can be written as a safe DRC query, and vice versa. They have exactly the same expressive power. This is a proven mathematical theorem, not an assumption. This equivalence justifies calling both "relational" query languages.

Example 3 — SQL goes beyond

SQL adds: aggregate functions (COUNT, SUM, AVG, MAX, MIN), GROUP BY, HAVING, ORDER BY, OUTER JOINS. None of these are expressible in pure relational algebra or calculus. SQL is a superset — relationally complete PLUS much more.

SQL: Queries, Aggregates, Constraints, Triggers

SELECT-FROM-WHERE, Nested queries, GROUP BY, HAVING, NULL, Integrity Constraints, Triggers

Basic SQL Query — SELECT-FROM-WHERE Slides 114–120

SQL is the language you use to talk to a relational database. The basic query structure is:

SELECT [DISTINCT] target-list -- which columns (projection) FROM relation-list -- which tables WHERE qualification -- which rows (selection)

The conceptual evaluation: compute the cross-product of all tables in FROM, discard rows failing the WHERE condition, project to just the SELECT columns, optionally eliminate duplicates with DISTINCT. The optimizer finds a much faster actual execution, but the result is the same.

Examples
Example 1 — Simple single-table query
SELECT S.sname, S.age FROM Sailors S WHERE S.rating > 7;

Finds names and ages of sailors with rating above 7. S is a range variable (alias) for Sailors. The DBMS scans Sailors, keeps rows where rating > 7, returns just sname and age columns.

Example 2 — Multi-table join query
SELECT S.sname FROM Sailors S, Reserves R WHERE S.sid = R.sid AND R.bid = 103;

Find names of sailors who reserved boat 103. Cross-product of Sailors and Reserves (all row pairs), keep rows where sids match AND bid=103, return snames. The WHERE S.sid=R.sid is the join condition; R.bid=103 is the filter.

Example 3 — String matching with LIKE
SELECT S.age, S.age - 5 AS age1, 2 * S.age AS age2 FROM Sailors S WHERE S.sname LIKE 'B_%B';

Find sailors whose name starts with B, ends with B, and has at least 3 characters (the underscore _ matches exactly one character, % matches zero or more). Also shows computed columns using arithmetic and column aliasing with AS.

Set Operations and Nested Queries Slides 121–127

SQL has set operations (UNION, INTERSECT, EXCEPT) and powerful nested queries (subqueries).

Examples
Example 1 — UNION: reserved red OR green boat
SELECT S.sid FROM Sailors S, Boats B, Reserves R WHERE S.sid=R.sid AND R.bid=B.bid AND B.color='red' UNION SELECT S.sid FROM Sailors S, Boats B, Reserves R WHERE S.sid=R.sid AND R.bid=B.bid AND B.color='green';

UNION automatically removes duplicates — a sailor who reserved both a red AND green boat appears only once. Use UNION ALL to keep duplicates.

Example 2 — Nested query with IN
SELECT S.sname FROM Sailors S WHERE S.sid IN ( SELECT R.sid FROM Reserves R WHERE R.bid = 103 );

The inner query (subquery) runs first, producing a list of sids who reserved boat 103. The outer query finds sailors whose sid is IN that list. Equivalent to the join version, but sometimes clearer to read.

Example 3 — Correlated subquery with EXISTS
SELECT S.sname FROM Sailors S WHERE EXISTS ( SELECT * FROM Reserves R WHERE R.bid = 103 AND S.sid = R.sid );

EXISTS checks if the subquery returns at least one row. The subquery references S.sid from the outer query — it's "correlated." For each sailor S, the subquery is re-run to check if they reserved boat 103. More expensive but sometimes necessary.

Aggregate Functions — COUNT, SUM, AVG, MAX, MIN Slides 128–132

Aggregate functions compute a single value from a set of rows. This extends relational algebra significantly — pure algebra cannot do aggregation.

  • COUNT(*) — count number of rows
  • COUNT(DISTINCT A) — count distinct values of column A
  • SUM(A) — total of column A
  • AVG(A) — average of column A
  • MAX(A) / MIN(A) — largest/smallest value in A
Examples
Example 1 — Simple aggregates
SELECT COUNT(*) -- how many sailors total? FROM Sailors S; SELECT AVG(S.age) -- average age of rating-10 sailors FROM Sailors S WHERE S.rating = 10; SELECT MAX(S.age), MIN(S.age) -- oldest and youngest FROM Sailors S;
Example 2 — Find name of oldest sailor
-- WRONG! Cannot mix aggregates and non-aggregate columns like this: -- SELECT S.sname, MAX(S.age) FROM Sailors S; -- CORRECT: Use a subquery SELECT S.sname, S.age FROM Sailors S WHERE S.age = (SELECT MAX(S2.age) FROM Sailors S2);

Why wrong? When you use MAX(age), the result is one number. But sname could be different for each row — which sname do you show? SQL rules: non-aggregate columns in SELECT must appear in GROUP BY, or be used in an aggregate function.

Example 3 — Aggregate operators can't be nested
-- WRONG: Cannot nest aggregates -- SELECT S.rating FROM Sailors S WHERE S.age = MIN(AVG(S2.age)...) -- CORRECT: Use a subquery/temp table SELECT Temp.rating FROM (SELECT S.rating, AVG(S.age) AS avgage FROM Sailors S GROUP BY S.rating) AS Temp WHERE Temp.avgage = (SELECT MIN(Temp2.avgage) FROM ... );

GROUP BY and HAVING — Grouping Results Slides 133–137

GROUP BY divides rows into groups based on shared values, then applies an aggregate to each group separately. HAVING filters groups (just as WHERE filters rows).

Full query form:

SELECT [DISTINCT] target-list FROM relation-list WHERE row-qualification -- filters rows BEFORE grouping GROUP BY grouping-list -- groups the remaining rows HAVING group-qualification -- filters groups AFTER grouping
⚠️
Critical rule for GROUP BY

Every column in SELECT that is NOT inside an aggregate function MUST appear in GROUP BY. Example: SELECT rating, AVG(age) → rating must be in GROUP BY. Because each group produces one output row — the DBMS needs a single value for each non-aggregate column (and only grouping columns guarantee that).

Examples
Example 1 — Average age per rating
SELECT S.rating, MIN(S.age) AS min_age FROM Sailors S WHERE S.age >= 18 GROUP BY S.rating;

Rows with age < 18 are excluded first (WHERE). Remaining rows are grouped by rating. For each rating group, MIN(age) is computed. One output row per rating level. rating must be in GROUP BY because it's in SELECT without an aggregate.

Example 2 — HAVING: filter groups with few members
SELECT S.rating, MIN(S.age) FROM Sailors S WHERE S.age > 18 GROUP BY S.rating HAVING COUNT(*) > 1;

Same as above but only output ratings that have MORE THAN ONE sailor (over age 18). HAVING filters groups after grouping, just as WHERE filters rows before. HAVING can use aggregate functions; WHERE cannot.

Example 3 — HAVING with subquery
SELECT S.rating, MIN(S.age) FROM Sailors S WHERE S.age > 18 GROUP BY S.rating HAVING 1 < (SELECT COUNT(*) FROM Sailors S2 WHERE S.rating = S2.rating);

HAVING can also contain a correlated subquery! This version: groups where the TOTAL count of sailors at that rating (including those ≤ 18) is > 1. Different from just counting those > 18.

NULL Values — The Three-Valued Logic Slide 138

NULL represents an unknown or inapplicable value. Null is NOT zero, NOT empty string, NOT false — it's the absence of a value. It complicates everything because any comparison with NULL is "UNKNOWN" (not true, not false).

SQL uses three-valued logic: TRUE, FALSE, UNKNOWN. WHERE clause keeps rows evaluating to TRUE. UNKNOWN rows are discarded.

  • NULL = NULL → UNKNOWN (not TRUE!). Use IS NULL or IS NOT NULL to check for nulls.
  • NULL > 5 → UNKNOWN. NULL + 5 → NULL.
  • TRUE AND UNKNOWN → UNKNOWN. FALSE AND UNKNOWN → FALSE. TRUE OR UNKNOWN → TRUE.
Examples
Example 1 — NULL in comparisons

Sailor's rating is NULL (not yet assigned). Query: WHERE rating > 8. For this sailor: NULL > 8 = UNKNOWN → row is discarded. Query: WHERE rating IS NULL → TRUE for this sailor → row is kept. Never write WHERE rating = NULL — that always evaluates to UNKNOWN!

Example 2 — NULLs in aggregates

COUNT(*) counts all rows including those with NULLs. COUNT(column) counts only non-NULL values in that column. AVG(salary) ignores NULL salaries — it computes the average of non-NULL salaries only. This can give surprising results if you're not careful.

Example 3 — Outer joins (concept)

Regular join: if a sailor has no reservations, they don't appear in Sailors ⋈ Reserves. LEFT OUTER JOIN: include all sailors even if they have no reservations — their reservation columns become NULL. This is where NULL becomes especially important: NULLs in outer join results represent "no matching row."

Integrity Constraints and Triggers Slides 139–144

Integrity Constraints (ICs) are conditions that every legal database instance must satisfy. The DBMS automatically enforces them — any INSERT, UPDATE, or DELETE that would violate an IC is rejected. Types:

  • Domain constraints: Column values must be of the right type (always enforced).
  • Primary key constraints: No two rows share the same primary key value.
  • Foreign key constraints: Foreign key values must exist in the referenced table.
  • General constraints (CHECK): Any condition you specify, e.g., rating must be 1–10.
  • ASSERTION: A constraint not attached to any single table, spanning multiple tables.

A Trigger is a stored procedure that fires automatically when a specified event (INSERT/UPDATE/DELETE) occurs on a table. It has three parts: Event (what triggers it), Condition (should it run?), Action (what it does).

Examples
Example 1 — CHECK constraint
CREATE TABLE Sailors ( sid INTEGER, sname CHAR(10), rating INTEGER, age REAL, PRIMARY KEY (sid), CHECK (rating >= 1 AND rating <= 10) );

Any attempt to insert a sailor with rating=11 or rating=0 is automatically rejected by the DBMS. You don't need to write application code to enforce this — the database handles it.

Example 2 — ASSERTION for cross-table constraint
CREATE ASSERTION smallClub CHECK ( (SELECT COUNT(S.sid) FROM Sailors S) + (SELECT COUNT(B.bid) FROM Boats B) < 100 );

Enforces: total sailors + total boats must be less than 100. This constraint spans two tables — you can't put it as a CHECK on just one table (because adding boats or sailors can both violate it).

Example 3 — Trigger: auto-populate view on insert
CREATE TRIGGER youngSailorUpdate AFTER INSERT ON Sailors REFERENCING NEW TABLE NewSailors FOR EACH STATEMENT INSERT INTO YoungSailors (sid, name, age, rating) SELECT sid, name, age, rating FROM NewSailors N WHERE N.age <= 18;

After any INSERT into Sailors, this trigger automatically fires and inserts qualifying young sailors into a YoungSailors table. Event = AFTER INSERT ON Sailors. Condition = age ≤ 18. Action = INSERT into YoungSailors. Triggers automate housekeeping without changing application code.

📋 SQL Summary (Slides 1–145)

  • SELECT-FROM-WHERE: Core query. WHERE = selection, SELECT = projection, FROM = relations to join.
  • DISTINCT: Eliminates duplicate rows from result.
  • UNION / INTERSECT / EXCEPT: Set operations on query results. Need union-compatible schemas.
  • Subqueries: Queries inside queries. Can use IN, EXISTS, ANY, ALL. Correlated subqueries re-run for each outer row.
  • Aggregates: COUNT, SUM, AVG, MAX, MIN. Compute one value per group. Cannot nest aggregates.
  • GROUP BY: Divides rows into groups. Non-aggregate SELECT columns must be in GROUP BY.
  • HAVING: Filters groups (like WHERE but for groups). Can use aggregates.
  • NULL: Unknown/inapplicable value. Three-valued logic. Use IS NULL to test.
  • ICs: Domain, primary key, foreign key, CHECK, ASSERTION. DBMS enforces automatically.
  • Triggers: Event-Condition-Action. Run automatically when data changes.
