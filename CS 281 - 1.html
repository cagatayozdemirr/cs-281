Schema Refinement & Normal Forms

Functional Dependencies, BCNF, 3NF, Decomposition

The Problem: Redundancy Slide 146

Imagine you're building a database for a company. You have one big table with employee information. Redundancy means the same piece of information is stored multiple times. This sounds harmless, but it causes serious problems called anomalies.

There are three kinds of anomalies:

  • Update anomaly: You have to update the same fact in multiple rows. If you forget one, your data is inconsistent.
  • Insert anomaly: You can't add some information without also adding other unrelated information.
  • Deletion anomaly: Deleting a row accidentally destroys information you still needed.
Examples
EXAMPLE 1 — Update Anomaly

You have a table: Employee(SSN, Name, Department, DeptPhone). The phone for the "Engineering" department is stored in every Engineering employee's row. If the phone number changes, you must update 50 rows. If you miss even one, your database has two different phone numbers for the same department — a contradiction.

EXAMPLE 2 — Insertion Anomaly

Same table. You want to record that a new department "Marketing" exists with phone 555-1234, but you have no employees yet. You can't insert the department because every row requires an employee SSN. The fact gets lost.

EXAMPLE 3 — Deletion Anomaly

The slides' own example: Hourly_Emps(SSN, Name, Lot, Rating, HrlyWage, HrsWorked) where Rating → HrlyWage (rating determines the wage). If you delete the only employee with Rating=5, you lose the information that "Rating 5 pays $7/hour" forever.

💡
The fix: Decomposition

Split the big table into smaller tables. Instead of one SNLRWH table, have SNLRH (employee data) and RW (rating → wage mapping). Now the wage for Rating 5 lives in RW and won't be lost when an employee is deleted.

Functional Dependencies (FDs) Slides 147–150

A Functional Dependency is a rule that says: "if two rows agree on these columns, they must agree on those other columns." Written as X → Y, pronounced "X determines Y".

Think of it like: knowing X gives you Y for free — you don't even need to look at any other column.

📐
Formal definition

X → Y holds over relation R if: for every two tuples t1 and t2 in R, whenever t1 and t2 have the same X values, they also have the same Y values. There are no exceptions allowed.

Examples
EXAMPLE 1 — SSN → Name (the classic)

In an employee table, your Social Security Number uniquely identifies you. So: SSN → Name. Two rows with the same SSN must have the same name. (One person can't have two names in the same database.)

EXAMPLE 2 — Rating → HrlyWage (the slides' example)

The company pays everyone with the same performance rating the same hourly wage. So Rating → HrlyWage. If two employees both have Rating=8, they must both earn $10/hour. This FD causes the redundancy problem above.

EXAMPLE 3 — ZipCode → City

In a US address table: ZipCode → City. Every 5-digit zip code belongs to exactly one city. Two rows with ZipCode=90210 must both say City="Beverly Hills". But City does NOT determine ZipCode (Beverly Hills could span multiple zips).

EXAMPLE 4 — Candidate Key

A candidate key is an attribute (or set) that determines all other attributes. If SSN is a key for Employees(SSN, Name, Dept, Salary), then SSN → SSN, Name, Dept, Salary — SSN determines everything. But SSN alone must be minimal: no proper subset of {SSN} should also determine everything.

Reasoning About FDs — Armstrong's Axioms Slides 151–153

If you know some FDs, you can derive new ones using three rules called Armstrong's Axioms. These are proven to be both correct (sound) and complete.

RuleWhat It SaysMemory Trick
ReflexivityIf Y is a subset of X, then X → Y. E.g., {SSN, Name} → {Name}."A superset always determines its subsets." Obvious stuff.
AugmentationIf X → Y, then XZ → YZ (add same thing to both sides)."Adding the same column to both sides preserves the dependency."
TransitivityIf X → Y and Y → Z, then X → Z.Like math: if A=B and B=C, then A=C.

Two useful derived rules:

  • Union: If X→Y and X→Z, then X→YZ (X determines both, so it determines their union).
  • Decomposition: If X→YZ, then X→Y and X→Z separately.
Examples
EXAMPLE 1 — Transitivity

Given: SSN → DeptID (employee belongs to one dept) and DeptID → ParkingLot (each dept has one lot). By transitivity: SSN → ParkingLot. We can derive where an employee parks just from their SSN.

EXAMPLE 2 — Augmentation

Given: Rating → Wage. By augmentation: {Rating, SSN} → {Wage, SSN}. Adding SSN to both sides doesn't break the dependency — it's just a trivial extension.

EXAMPLE 3 — Chaining from the slides

Contracts(cid, sid, jid, did, pid, qty, value). Given: JP→C and C→CSJDPQV (C is the key). By transitivity: JP→CSJDPQV. Also: SD→P implies by augmentation: SDJ→JP. Then by transitivity: SDJ→CSJDPQV.

Attribute Closure — Checking if an FD Holds Slide 153

Given a set of FDs, how do you check whether a new FD like A→E is also implied? You compute the attribute closure of A, written A⁺. This is "all attributes that A can determine, directly or by transitivity."

Algorithm: Start with the set {A}. Repeatedly: if you see any FD whose left side is already in your set, add its right side. Stop when nothing new can be added.

Examples
EXAMPLE 1 — From the slides: Does A→E hold?

FDs: A→B, B→C, CD→E. Does A→E?
Start: A⁺ = {A}
A→B fires: A⁺ = {A, B}
B→C fires: A⁺ = {A, B, C}
CD→E: we have C but not D, so it can't fire yet. E is not in A⁺.
Answer: No, A→E does NOT hold with these FDs alone.

EXAMPLE 2 — Is {SSN} a key?

Table: Employee(SSN, Name, Dept, Salary). FDs: SSN→Name, SSN→Dept, SSN→Salary.
Compute SSN⁺: start {SSN}. Add Name, Dept, Salary. SSN⁺ = {SSN, Name, Dept, Salary}.
Since SSN⁺ = all attributes, SSN is a candidate key.

EXAMPLE 3 — A more complex closure

FDs: A→BC, B→D, CD→E. Compute A⁺.
Start: {A} → A→BC fires → {A,B,C} → B→D fires → {A,B,C,D} → CD→E fires → {A,B,C,D,E}.
So A determines everything: A is a key for ABCDE.

Boyce-Codd Normal Form (BCNF) Slides 154–155

BCNF is the "ideal" normal form. A relation is in BCNF if every non-trivial functional dependency has a superkey on its left side. In other words: the only things that "determine" other things should be keys (or superkeys). No column should determine another column unless it's a key.

📋
Formal rule: R is in BCNF if, for every FD X→A in F⁺:

Either (1) A is in X (trivial FD — A is part of the left side), OR (2) X is a superkey of R.

Examples
EXAMPLE 1 — BCNF Violation

Table: SNLRWH (SSN, Name, Lot, Rating, Wage, Hours). Key is S (SSN). FD: R→W (Rating determines Wage). Is this BCNF? R is NOT a superkey (it doesn't uniquely identify employees). So R→W violates BCNF. Fix: split into SNLRH and RW.

EXAMPLE 2 — BCNF-Compliant Table

After splitting: Wages(Rating, HrlyWage) with key=Rating. The only non-trivial FD here is Rating→HrlyWage, and Rating IS the key. So this table IS in BCNF. ✓

EXAMPLE 3 — Student Courses

Table: StudentCourse(StudentID, CourseID, InstructorID). Say each course has exactly one instructor: CourseID→InstructorID. If the key is {StudentID, CourseID}, then CourseID alone is not a key — BCNF violated! Fix: split into Course(CourseID, InstructorID) and Enrollment(StudentID, CourseID).

Third Normal Form (3NF) — The Compromise Slides 156–158

Sometimes BCNF decomposition forces you to lose information about FDs (you can't check them without doing a join). 3NF is a slightly weaker condition that avoids this. It allows one extra escape: a column A can be "determined by" a non-key X, as long as A is part of some candidate key of the relation.

⚠️
3NF Rule: R is in 3NF if, for every FD X→A in F⁺:

Either (1) trivial FD, OR (2) X is a superkey, OR (3) A is part of some candidate key of R.

The key insight: BCNF implies 3NF, but not vice versa. 3NF allows a little redundancy but guarantees that a lossless-join, dependency-preserving decomposition always exists.

Examples
EXAMPLE 1 — 3NF but not BCNF

Table: Reserves(SailorID, BoatID, DayID, CreditCardID). FDs: S→C (sailor has one card) and C→S (card belongs to one sailor). Keys are {S,B,D} and {C,B,D}. The FD S→C violates BCNF (S is not a key). But C is part of key {C,B,D}, so 3NF is satisfied. This table stays in 3NF even though there's some redundancy.

EXAMPLE 2 — Why 3NF allows redundancy

In the Reserves example, for every reservation of sailor S, we store the (S,C) pair again. If a sailor has 10 reservations, we redundantly store their credit card 10 times. That's the price of 3NF vs BCNF.

EXAMPLE 3 — When to use 3NF

If decomposing to BCNF would lose a critical FD (making it impossible to enforce without joining tables), use 3NF instead. It's a practical compromise: some redundancy in exchange for being able to check all constraints locally within each table.

Decomposition: Lossless-Join & Dependency-Preserving Slides 159–166

When we split a table R into smaller tables, we need two guarantees:

  1. Lossless-Join: When we join the pieces back together, we get exactly the original data — not more, not less. No "spurious tuples" (fake rows that weren't there originally).
  2. Dependency-Preserving: Every original FD can still be checked by looking at just one of the smaller tables. We don't need to do a join to enforce any constraint.
✅
Lossless-join test

Decomposing R into X and Y is lossless-join if and only if: (X∩Y) → X, OR (X∩Y) → Y. That is, the shared columns between the two tables determine at least one of them entirely.

Examples
EXAMPLE 1 — Lossless decomposition

Table SNLRWH, FDs include R→W. Decompose into SNLRH and RW. Shared column: R. Does R→RW? Yes (R determines W, and trivially R→R). So this IS lossless. Joining on R gives back the original table. ✓

EXAMPLE 2 — Lossy (bad) decomposition

Table: ABC with no FDs. Decompose into AB and BC. Shared column is B. B does NOT determine A or C (no FDs exist). Joining AB and BC on B produces extra rows — spurious tuples. This is a lossy decomposition. Bad! ✗

EXAMPLE 3 — Dependency preservation failure

Table CSJDPQV, key=C, FDs: JP→C, SD→P, J→S. BCNF decomposition gives: SDP, JS, CJDQV. The FD JP→C involves J (in CJDQV) and P (in SDP but not CJDQV). To check JP→C, you must join CJDQV with SDP. This decomposition is not dependency-preserving. A solution: add table JPC explicitly just for enforcement.

BCNF Decomposition Algorithm Slides 167–169

To decompose R into BCNF: find any FD X→Y that violates BCNF (X is not a superkey). Replace R with two relations: (R − Y) and XY. Repeat until all relations are in BCNF.

BCNF Decomposition Algorithm: Input: Relation R, set of FDs F 1. If R is already in BCNF, done. 2. Find a violating FD X→Y (X is not a superkey of R). 3. Split R into: R1 = R − Y (everything except Y) R2 = X ∪ Y (the violating dependency becomes its own table) 4. Recurse on R1 and R2 with their projected FDs.
Examples
EXAMPLE 1 — Simple decomposition

SNLRWH, key=S, violating FD: R→W. Split:
• R1 = SNLRH (remove W from main table)
• R2 = RW (the offending dependency becomes its own table)
Both R1 (key=S) and R2 (key=R) are now in BCNF.

EXAMPLE 2 — Multi-step decomposition

CSJDPQV, key=C, violating FDs: SD→P, J→S.
Step 1: SD→P violated. Split into SDP and CSJDQV.
Step 2: J→S violates BCNF in CSJDQV. Split into JS and CJDQV.
Final: SDP, JS, CJDQV — all in BCNF.

EXAMPLE 3 — Order matters!

With multiple violating FDs, the order in which you pick them affects the final decomposition. Different choices can yield very different sets of tables. This is not a bug — it's expected. The algorithm guarantees BCNF and lossless-join, but not always dependency preservation.

🗺 Schema Refinement Summary

  • Redundancy causes update, insert, and deletion anomalies — solve with decomposition.
  • FDs formally capture which columns determine which others.
  • BCNF = only keys can determine non-trivial attributes. Best at eliminating redundancy.
  • 3NF = slightly weaker; always allows lossless-join + dependency-preserving decomposition.
  • Always aim for lossless-join. Dependency preservation is desired but sometimes impossible with BCNF.
  • Minimal cover: smallest set of FDs that captures the same closure — used for 3NF decomposition.
Chapter 10 · Slides 170–195

Overview of Storage & Indexing

File Organizations, Indexes, Cost Analysis

Data on External Storage — Why It Matters Slide 171

Databases store data on disks (external storage), not in RAM. This is the fundamental performance challenge. Disks are thousands of times slower than RAM. Every design decision in a database is driven by minimizing disk reads/writes.

  • Disks: You can jump to any page at a fixed cost (random access). But reading several consecutive pages is much cheaper than jumping around randomly — sequential reads are faster.
  • Tapes: Even cheaper but you can only read sequentially. Used for backups and archives.
  • Buffer Pool: The DBMS keeps frequently-used pages in RAM (a "buffer pool") to avoid re-reading from disk.
Examples
EXAMPLE 1 — Analogy: Library Books

Think of the disk as a huge library. RAM is your desk. It's fast to work with books on your desk, slow to walk to the library and back. The buffer pool = the books currently on your desk. A database tries to keep the books you'll need next on your desk already.

EXAMPLE 2 — Sequential vs. Random

Reading 1,000 consecutive pages from disk: fast (disk head moves forward smoothly). Reading 1,000 random pages scattered across the disk: extremely slow (disk head must seek back and forth). A full table scan (reading the whole file) is often faster than random access for the same data, which is why clustering matters so much.

EXAMPLE 3 — Record ID (rid)

Every record on disk gets a unique identifier called a rid (record ID), which encodes which page and which slot within that page. An index stores rids so it can say "here's where to find this record on disk" without scanning the whole file.

Three Ways to Organize Files Slides 172–173

There are three main ways to physically arrange records on disk. Each has different performance for different operations:

OrganizationWhat It IsBest ForWorst For
Heap File Records are inserted in any order (end of file). No sorting. Full table scans — reading everything Finding specific records quickly
Sorted File Records are kept sorted by some field (e.g., age). Like a sorted phone book. Range queries ("all employees age 30–40") Insertions (must maintain sort order)
Index File A separate structure (B+ tree or hash table) that points to records by key value. Selective searches, updates Slight overhead on updates
Examples
EXAMPLE 1 — Heap File

An employee HR table where you always SELECT * (dump all records). A heap file is perfect — just scan from start to end. No structure needed. Inserting a new employee is also instant: just append to the end of the file.

EXAMPLE 2 — Sorted File

A bank's transaction log sorted by date. "Show me all transactions from January" is a range query. Binary search finds January 1st quickly, then you scan forward. But inserting a transaction from 3 years ago means shifting many records — expensive!

EXAMPLE 3 — Index

An index on employee ID in a company database. "Find employee 47382" uses the index to jump directly to that record's disk location — you don't scan the whole file. Much faster for point lookups, and updates are much easier than maintaining a sorted file.

What Does an Index Store? — Three Alternatives Slides 174–176

An index stores "data entries" k* for each key k. There are three ways to define what k* means:

AlternativeWhat k* ContainsTrade-off
Alt 1The full data record itself (the index IS the file)Only one such index possible. Large entries.
Alt 2<key, rid> — key + pointer to one recordSmall entries. Good if duplicates are rare.
Alt 3<key, list of rids> — key + list of pointersEven more compact for duplicates, but variable-size entries.
Examples
EXAMPLE 1 — Alt 1: Phone book analogy

The phone book itself IS the index (sorted by name, contains full address). You can't have two phone books sorted differently with the full address in each — that's duplicate storage. That's why Alt 1 can only be used for one index per table.

EXAMPLE 2 — Alt 2: Card catalog in old library

The card catalog has an index card saying "Shakespeare, William → Shelf B7, Row 3." The card is tiny (just the name and location). The actual book is elsewhere. This is Alt 2: the index entry is <author_name, page_id>.

EXAMPLE 3 — Alt 3: Multiple locations

If many books have the same author (like Stephen King wrote 50 books), Alt 3 stores: <"Stephen King", [shelf1, shelf4, shelf7, ...]>. One entry, many locations. More compact than Alt 2 (which would have 50 separate entries), but the entry is variable-length.

Clustered vs. Unclustered Indexes Slides 177–178

This is one of the most important distinctions in indexing and a likely exam topic!

  • Clustered Index: The physical order of records on disk matches the order of the index. Like a sorted file. Data entries and data records are "in sync." Only ONE clustered index per table is possible.
  • Unclustered Index: The index is sorted, but the actual records on disk are in a different order. You might have to jump all over the disk to retrieve matching records.
⚠️
Why clustering matters enormously for range queries

For a clustered index on age, "find all employees aged 30–40" reads a contiguous block of disk — one sequential scan. For an unclustered index, each matching record might be on a different page — potentially one disk I/O per record!

Examples
EXAMPLE 1 — Clustered: Dictionary

A dictionary is clustered on the word. Words appear physically in alphabetical order. Finding all words starting with "pre-" is fast — just flip to the P section and scan a few pages forward. The physical layout matches the logical order.

EXAMPLE 2 — Unclustered: Book index

The index at the back of a textbook lists topics alphabetically, but the referenced pages are scattered: "Binary trees: pages 34, 89, 156, 203." You must flip back and forth across the book. Unclustered indexes are like this — the index is sorted, but your data jumps around disk.

EXAMPLE 3 — Only one clustered index

Employee table. You can cluster on either DepartmentID or Salary, but not both simultaneously. If clustered on DeptID, then "find all employees in Dept 5" is fast (all Dept 5 employees are together). But "find all employees earning $50K–$60K" is slow — their records are scattered.

Cost Analysis — How We Compare File Organizations Slides 180–186

The course uses a simplified model to compare performance. We count only disk I/O page reads/writes, ignoring CPU cost.

Variables:

  • B = number of data pages in the file
  • R = records per page
  • D = time to read or write one disk page (in ms)

Examples to understand the table
EXAMPLE 1 — Heap: Equality Search = 0.5BD

To find one specific record in an unsorted heap, you scan until you find it. On average you find it halfway through — so you read 0.5 × B pages, costing 0.5BD. In the worst case it's the last record (BD). Best case it's first (D).

EXAMPLE 2 — Sorted file: Equality = D·log₂B

Binary search! With B pages, you need log₂B comparisons to find the right page, each costing D. If B=1024 pages, log₂(1024) = 10 page reads vs. 512 for heap. Huge win for equality. But inserting requires shifting pages to maintain order — costs BD.

EXAMPLE 3 — Unclustered index: worst case

If a range query returns 1,000 matching records scattered across the disk, and each record is on a different page, you need 1,000 page reads — one per matching record. This is why unclustered indexes are terrible for range queries on large result sets. Cost = number of matching records × D.

Choosing the Right Index Slides 187–194

The choice of which index to build is crucial for performance. Key guidelines:

  • Attributes in WHERE clauses are candidates for index keys.
  • Exact match (=) queries → Hash index is best (constant time lookup).
  • Range queries → Tree index (B+ tree) is best. Hash can't support ranges!
  • One index per table can be clustered — choose the one that helps the most important range queries.
  • Composite index: if WHERE has multiple conditions, a multi-attribute key index can help.
🎯
Index-only plans

Sometimes a query can be answered using ONLY the index, without reading the actual data records. If SELECT asks only for columns that are all in the index key, the DBMS never touches the data file. This is extremely fast. Example: index on (dept, salary), query SELECT dept, COUNT(*) GROUP BY dept — only the index is needed.

Examples
EXAMPLE 1 — Choose clustered for range queries

Query: "Find all employees aged 30–50." There are many such employees. A clustered B+ tree on age reads them all in one sequential scan. An unclustered index would need one disk I/O per employee — potentially thousands of random reads.

EXAMPLE 2 — Composite key ordering matters

WHERE age=30 AND salary > 40000. Index on <age, salary>: find all age=30 entries, then scan by salary (sorted within age). Index on <salary, age>: finds salary range first, then you must check each for age=30. The first order is much better for this query.

EXAMPLE 3 — Index maintenance overhead

Every time you INSERT, UPDATE, or DELETE, every index on that table must be updated too. A table with 5 indexes pays 5× the update cost vs. a heap. Choose indexes carefully — only build them if the query speedup outweighs the update overhead.

Chapter 11 · Slides 196–221

Tree-Structured Indexes

ISAM, B+ Trees, Insert, Delete, Bulk Loading

Why Tree Indexes? Range Searches Slides 197–198

Suppose you want to find all students with GPA above 3.0. If the data is sorted, you can binary search for the first student with GPA 3.0, then scan forward. But binary search on a large file still needs log₂(N) disk reads. The solution: build an index file — a smaller sorted structure that points into the data file.

Apply the idea recursively: build an index on the index — that's a tree!

Examples
EXAMPLE 1 — The index idea

Data file: 1 million pages. Index file: 1,000 pages (one entry per data page). To find a record, binary search the index (10 reads), then go to the data page (1 read) = 11 reads total vs. ~500,000 reads for a full scan.

EXAMPLE 2 — Range search with tree

"Find all employees with salary 50,000–70,000." Tree traversal finds the leaf with salary=50,000, then you scan leaf pages forward (they're linked!) until you pass 70,000. This is one of the key advantages of B+ trees over hash indexes.

EXAMPLE 3 — ISAM vs B+ tree preview

ISAM (Index Sequential Access Method) is a static tree — great when data doesn't change. B+ tree is dynamic — it automatically re-balances when records are inserted or deleted. In real databases, B+ trees win because data always changes.

ISAM — The Static Tree Index Slides 199–202

ISAM works by building a tree at creation time. Leaf pages contain the actual data entries. Non-leaf pages contain index entries that guide your search downward. The structure is static — the tree itself never changes after creation. Only leaf pages get modified (via overflow pages for extra insertions).

ISAM tree structure: Root | [40] / \ [20][33] [51][63] / | | \ Leaf Leaf Leaf Leaf ← Primary pages (fixed at creation) | [overflow] ← Overflow page (created when leaf is full)

Problem: As you insert many records, overflow chains grow long and performance degrades. The tree itself never gets re-balanced.

Examples
EXAMPLE 1 — Search in ISAM

Find record 37 in the tree above. Start at root: 37 < 40, go left child. Node has 20 and 33. 20 < 37 and 33 < 37, go to rightmost child. That leaf page contains records 33–39. Scan to find 37. Total cost: height of tree + 1 page.

EXAMPLE 2 — Insert causing overflow

Insert records 23, 48, 41, 42 into the tree. They go to the appropriate leaf pages. If a leaf is full, an overflow page is chained to it. The tree structure (non-leaf pages) stays the same — only the leaf level changes. After many inserts, searching requires checking overflow pages, slowing things down.

EXAMPLE 3 — Delete in ISAM

Delete record 42. Find it in the overflow page, remove it. If the overflow page becomes empty, de-allocate it. The non-leaf pages are NEVER changed, even if this means the non-leaf page says "51" but 51 doesn't exist anymore at the leaf level — a ghost entry. (ISAM tolerates this.)

B+ Trees — The Dynamic Tree Index Slides 203–207

A B+ tree is like ISAM but self-balancing. Every time you insert or delete, the tree adjusts to maintain balance. The height is always O(logF N) where F is the fanout (number of pointers per node) and N is the number of leaf pages.

Key properties:

  • Leaf pages: Contain actual data entries (k* values). They are doubly-linked left↔right for range scans.
  • Non-leaf pages: Contain only keys to guide search. Each node has pointers between keys: P₀ | K₁ | P₁ | K₂ | P₂ | ... | Kₘ | Pₘ
  • Minimum 50% occupancy: Each node must be at least half full (except the root). If a node has order d, it holds between d and 2d keys.
  • Fanout: A typical B+ tree has fanout F ≈ 100–200, meaning 3–4 levels handles 100+ million records!
B+ Tree Example: Root ┌──[17]──┐ / \ [5][13] [24][30] / | \ / | \ [2*3*][5*7*8*][14*16*][19*20*22*][24*27*29*][33*34*38*39*] └───────────────────────────────────────────────┘ Leaves are linked →→→ for range scans
Examples
EXAMPLE 1 — Searching

Search for 15* in the tree above. Start at root: 15 ≤ 17, go left. Node has 5 and 13. 15 > 13, go right subtree. Arrive at leaf [14* 16*]. Scan: 14 → 15 not there! 15 is NOT in the tree. B+ trees can tell you a record doesn't exist without scanning the whole file.

EXAMPLE 2 — Range search "all entries ≥ 24"

Navigate to leaf containing 24*. Then follow the next-leaf pointers: [24*27*29*] → [33*34*38*39*] → ... Read all leaves from 24 onward in one sequential scan. This is extremely efficient — no backtracking needed.

EXAMPLE 3 — Practical capacity

A B+ tree with order 100 and 67% fill factor has average fanout 133. Height 4: 133⁴ ≈ 312 million records, found in 4 page reads. Height 3: 133³ ≈ 2.3 million records in 3 reads. Plus: level 1 (1 page) and level 2 (133 pages = 1 MB) often fit in RAM — meaning only 1–2 actual disk reads needed for most searches!

Inserting Into a B+ Tree Slides 208–211

Insertion in a B+ tree follows these steps:

  1. Find the correct leaf page L (same as a search).
  2. If L has space, insert and done.
  3. If L is full: split L into two pages (L and L2). Redistribute entries evenly. Copy up the middle key into the parent (the middle key stays in both the leaf and the parent).
  4. The parent may now be overfull — repeat recursively. If a non-leaf page splits, push up (not copy up) the middle key — it moves out of the node and goes only to the parent.
  5. If the root splits, a new root is created and the tree height increases by 1.
🔑
Copy-up vs Push-up

Leaf split → COPY UP the middle key (it stays in the leaf too, because leaves store data). Non-leaf split → PUSH UP the middle key (it only goes to parent, not kept in either child, because non-leaves only guide searches).

Examples
EXAMPLE 1 — Simple leaf insertion (no split)

Insert 6* into a leaf [5* 7* 8*] with order d=2 (capacity 2d=4). The leaf has 3 entries and can hold 4, so just insert: [5* 6* 7* 8*]. Done. O(log N) just to find the leaf.

EXAMPLE 2 — Leaf split (inserting 8* from the slides)

Insert 8* into a full leaf [2* 3* 5* 7*] (order 2, so max is 4 = full). Split: left half stays [2* 3*], right half [5* 7* 8*]. Middle key (5) is COPIED UP to parent. The parent now has 5 inserted into it and points to both new leaf pages.

EXAMPLE 3 — Root split increases height

After inserting 8*, the root [13 17 24 30] may become overfull. Split the root: left child gets [5 13], right child gets [24 30], middle key (17) PUSHES UP to a new root. The tree grows one level taller. This is how B+ trees grow — always from the top.

Deleting From a B+ Tree Slides 212–216

Deletion must maintain the half-full invariant. Steps:

  1. Find leaf L containing the entry, remove it.
  2. If L is still at least half-full: done.
  3. If L has fewer than d entries (too empty): try to borrow from a sibling (same parent). If sibling has extra entries, redistribute and update parent key.
  4. If can't borrow: merge L with sibling. Pull down the separator key from parent. Delete the merged node from parent. If parent is now too empty, propagate up recursively.
  5. If root has only one child, remove root — tree height decreases.
Examples
EXAMPLE 1 — Easy delete (19*)

Delete 19* from leaf [19* 20* 22*] (order 2, so min occupancy is 2). After deletion: [20* 22*] — still 2 entries, still half-full. Done. No structural change needed.

EXAMPLE 2 — Re-distribution (delete 20*)

Delete 20* from [20* 22*] (order 2, min=2). After deletion: [22*] — only 1 entry, below minimum! Check sibling [24* 27* 29*] — it has 3 entries, can spare one. Borrow: move 24* to the left leaf, update the parent key to the new separator (which becomes 24, copied up). Result: [22* 24*] and [27* 29*].

EXAMPLE 3 — Merge (delete 24*)

Delete 24* from [22* 24*]. After deletion: [22*] — too empty. Check sibling [27* 29*] — also at minimum (2), can't spare. So merge: combine [22*] and [27* 29*] into one leaf [22* 27* 29*]. Pull down the separator from parent. Parent may now be too empty — propagation happens upward. If root shrinks to one child, it's removed and tree height decreases.

Bulk Loading — Building a B+ Tree Fast Slides 217–219

Inserting millions of records one at a time into a B+ tree is very slow — each insert traverses the tree and may cause splits. Bulk loading builds the tree from scratch much faster:

  1. Sort all data entries by the search key.
  2. Fill leaf pages left-to-right from sorted data.
  3. Build index pages above, always inserting the first key of each new leaf into the rightmost index page above it.
  4. When an index page fills, split it and push up — same as insertion, but only along the rightmost path.
Examples
EXAMPLE 1 — Why faster?

Normal insertion: for 1 million records, each insert reads O(log N) pages. 1,000,000 × 4 reads = 4 million disk reads. Bulk loading: sort once (linear scan), then fill pages sequentially. Almost all writes are sequential — much cheaper. Fewer total page reads/writes.

EXAMPLE 2 — Sequential leaf pages

Bulk loading produces leaves that are stored sequentially on disk. This makes future range scans faster (sequential I/O). Repeated single-row inserts scatter leaf pages all over disk — worse for range scans even if the tree is logically the same.

EXAMPLE 3 — Fill factor control

Bulk loading lets you control how full each leaf is packed ("fill factor"). If you expect many future inserts, set fill factor to 70% — leaving room so future inserts don't immediately cause splits. If the data is read-only, pack to 100% for maximum space efficiency.

Hash-Based Indexes

Static Hashing, Extendible Hashing, Linear Hashing

How Hash Indexes Work Slides 223–225

A hash-based index uses a hash function to map a key to a bucket number. The record is stored in (or pointed to by) that bucket. Lookup is O(1) — find the bucket, check if record is there. No scanning needed.

Critical limitation: Hash indexes support only equality searches (find WHERE key = 42). They CANNOT support range queries (WHERE key > 42) because related keys are scattered randomly across buckets.

Static Hashing: key "Alice" ──→ h("Alice") mod M = bucket 3 key "Bob" ──→ h("Bob") mod M = bucket 7 key "Carol" ──→ h("Carol") mod M = bucket 3 (same bucket as Alice!) Bucket 3: [Alice's record | Carol's record | ... ] Bucket 7: [Bob's record | ... ] If bucket 3 fills up → overflow page chained on
Examples
EXAMPLE 1 — Hash function

h(key) = (a × key + b) mod M where M is number of buckets. E.g., M=10, a=3, b=7. Key=5: (3×5+7) mod 10 = 22 mod 10 = 2. Key=15: (3×15+7) mod 10 = 52 mod 10 = 2. Both go to bucket 2 — a collision. Good hash functions spread keys evenly to minimize collisions.

EXAMPLE 2 — Why hashing is great for equality

"Find employee with SSN = 123456789." Hash the SSN: h(123456789) = bucket 47. Go directly to bucket 47. Scan the (small) bucket. Done — typically 1–2 disk reads. Compare: full scan of a million-record heap file = thousands of reads.

EXAMPLE 3 — Why hashing fails for range queries

"Find all employees with salary between $50,000 and $60,000." With a hash index on salary, $50,001 hashes to bucket 3, $50,002 to bucket 8, $50,003 to bucket 1... There's no pattern. You must scan ALL buckets — just as bad as a full table scan. Use a B+ tree for ranges!

Static Hashing — The Overflow Problem Slide 225

In static hashing, the number of buckets M is fixed at creation. As more records are inserted, buckets fill up and require overflow pages — extra pages chained to the bucket. Long overflow chains slow down searches (you must scan the whole chain).

Examples
EXAMPLE 1 — Overflow chains grow

Bucket capacity: 4 records. Insert 20 records all hashing to bucket 5. Bucket 5 fills at 4, gets overflow page 1 (4 more), overflow page 2 (4 more), overflow page 3 (4 more), overflow page 4 (4 more). Now finding any record in bucket 5 requires reading 5 pages — as slow as a small scan.

EXAMPLE 2 — Why not just increase M?

Doubling M to fix overflow means reading and rehashing ALL records into new bucket locations. This is a massive operation on a live database. For a 100 GB database, it's impractical to do this every time performance degrades.

EXAMPLE 3 — Skewed data makes it worse

If your data is skewed (e.g., 90% of employees are in department 3), then h(dept=3) always hits one bucket while others are nearly empty. Static hashing can't adapt to this. Dynamic hashing methods (extendible, linear) solve this.

Extendible Hashing — Smart Dynamic Hashing Slides 226–231

Extendible Hashing avoids the overflow chain problem by using a directory of pointers to buckets. When a bucket overflows, you split only that one bucket (not the whole file). If necessary, the directory doubles in size — but since the directory is much smaller than the data, this is cheap.

The trick: use the last global depth bits of h(key) to index into the directory. Each bucket has a local depth — how many bits are used to distinguish records in that bucket.

📖
Global Depth vs Local Depth

Global depth: how many bits the directory currently uses. Directory has 2^(global depth) entries. Local depth: how many bits are used to decide if a record goes into this specific bucket. When local depth = global depth and a bucket splits, the directory must double.

Examples
EXAMPLE 1 — Basic lookup

Global depth = 2. Directory has 4 entries (00, 01, 10, 11). To find a record with h(key) = 5 = binary 101: take last 2 bits = "01". Directory entry 01 points to Bucket B. Scan Bucket B for the record. Total: 1 directory lookup + 1 bucket read.

EXAMPLE 2 — Insert causing bucket split (no directory doubling)

Insert h(r)=20 (binary 10100). Last 2 bits = "00" → Bucket A. Bucket A is full. Split Bucket A into A (keeps records ending in 000) and A2 (records ending in 100). Local depth of A and A2 increases to 3. Now directory entry "100" points to A2, "000" still to A. If global depth was 2, we need 3 bits → directory doubles from 4 to 8 entries.

EXAMPLE 3 — Why least significant bits?

Using least significant bits lets directory doubling work by simple copying. Old directory: [00→B0, 01→B1, 10→B2, 11→B3]. New directory (doubled): [000→B0, 001→B1, 010→B2, 011→B3, 100→B0, 101→B1, 110→B2, 111→B3]. The second half is just a copy of the first — each new entry points to the same bucket as its "lower half" counterpart. Super efficient!

Linear Hashing — Hashing Without a Directory Slides 232–237

Linear Hashing is another dynamic hashing approach, but it avoids the directory entirely. Instead, it uses overflow pages and splits buckets in a fixed round-robin order. This prevents long overflow chains from developing systematically.

Key ideas:

  • Use a family of hash functions h₀, h₁, h₂, ... where hᵢ(key) = h(key) mod (2ⁱ × N)
  • N = initial number of buckets. Each "round" doubles the effective number of buckets.
  • A pointer Next tracks which bucket to split next (always sequential).
  • When searching, apply hLevel(key). If result is before Next, also try hLevel+1 (the split image bucket).
Examples
EXAMPLE 1 — Search in Linear Hashing

Level=0, N=4, Next=1. Search for key with h(key)=5. h₀(5) = 5 mod 4 = 1. Is 1 ≥ Next(=1)? Yes — bucket hasn't been split yet in this round. But wait: 1 = Next means it's exactly the next to split. Check: if bucket 1 has been split, use h₁(5) = 5 mod 8 = 5. Otherwise use bucket 1. The exact rule: if h₀(r) < Next, use h₁; else use h₀.

EXAMPLE 2 — Overflow handling

A bucket overflows → add overflow page AND split the Next bucket (which may not be the overflowing one). This keeps overflow chains short across the whole structure, because splits happen evenly (round-robin), not concentrated on "hot" buckets.

EXAMPLE 3 — LH vs Extendible Hashing

Extendible Hashing: no overflow pages, directory can grow large for skewed data. Linear Hashing: no directory (saves memory), short overflow chains, handles duplicates easily. But LH splits may be wasteful (splitting a non-full bucket). In practice, both work well — LH is simpler to implement without a directory structure.

📊 Hash Index Summary

  • Hash indexes: ONLY for equality searches (WHERE key = value). Cannot do range queries.
  • Static hashing: fixed buckets, overflow chains can grow → performance degrades.
  • Extendible hashing: directory of pointers, splits one bucket at a time, doubles directory occasionally. Fast equality lookup (1–2 disk reads if directory fits in RAM).
  • Linear hashing: no directory, splits round-robin, short overflow chains. Handles duplicates. Simpler but potentially lower space utilization.
  • For range queries: always use B+ tree.
Chapter 13 · Slides 241–263

Transaction Management

ACID, Concurrency, Locks, Logging, Recovery

What Is a Transaction? Slides 242–244

A transaction (Xact) is the database's view of a user program: a sequence of reads and writes. From the database's perspective, it doesn't matter that you wrote 10,000 lines of Python — it only sees "read A, write A, read B, write B..."

Why do we need transactions? Because multiple users access the database at the same time (concurrency), and crashes can happen at any moment. Transactions give us guarantees about what happens in these scary situations.

🏦
Classic Example: Bank Transfer

T1: Transfer $100 from account B to account A: A = A + 100, B = B - 100.
T2: Apply 6% interest to both: A = 1.06 × A, B = 1.06 × B.
Both run at the same time. What could go wrong?

Examples
EXAMPLE 1 — Why concurrency is essential

A bank processes 10,000 transactions per second. If each transaction ran one at a time, and each takes 10ms (disk reads), you'd process 100/second — far too slow. Concurrency (interleaving many transactions) keeps the CPU and disk busy, achieving much higher throughput.

EXAMPLE 2 — The DBMS only sees reads and writes

Your code: "calculate interest, adjust for inflation, apply penalty fee, check fraud rules, update balance." The DBMS only sees: R(A), W(A), R(B), W(B). All the business logic is invisible to the DBMS — it just manages the read/write sequence safely.

EXAMPLE 3 — Consistent state

"The total money in the bank must always be the same after a transfer." This is a consistency rule. Each transaction must take the database from one consistent state to another. If T1 adds $100 to A and subtracts $100 from B — before or after, total money is the same. Halfway through T1 (A added but B not yet subtracted), the database is temporarily inconsistent — that's allowed during a transaction, but must be fixed by the end.

Atomicity — All or Nothing Slide 245

Atomicity is the guarantee that a transaction either completes ALL its steps, or it completes NONE of them. There's no "halfway done" from the outside world's perspective. If the system crashes while T1 is in progress, all of T1's changes are rolled back as if T1 never happened.

How? The DBMS maintains a log of every write. If a crash occurs, the log lets it undo incomplete transactions.

Examples
EXAMPLE 1 — Bank transfer atomicity

T1: A=A+100, B=B-100. If the system crashes after A is updated but before B is updated — A has $100 extra but B hasn't lost $100. Money was "created from nothing"! Atomicity prevents this: the crash-recovery system sees T1 was incomplete, rolls back A to its original value. As if T1 never ran.

EXAMPLE 2 — Commit vs Abort

A transaction commits when all steps are done and changes are permanent. It aborts (is rolled back) when something goes wrong. Once committed, changes survive even a crash. An aborted transaction leaves no trace. From a user's view, every transaction is atomic — either it ran completely, or it didn't run at all.

EXAMPLE 3 — Online shopping analogy

You order a product online. The transaction: (1) reserve item in inventory, (2) charge your card, (3) create shipping label. If step 2 (charge) fails, the system must un-reserve the item (rollback step 1) — atomicity. You can't end up with an item reserved and your card not charged, or vice versa.

Serializability — Correct Concurrency Slides 247–248

When multiple transactions run concurrently, their operations are interleaved. A schedule defines the order of all operations from all transactions. There are three kinds:

  • Serial schedule: T1 runs completely, then T2 runs completely. No interleaving. Always correct but slow.
  • Serializable schedule: An interleaved schedule that produces the SAME result as some serial execution. This is the goal!
  • Non-serializable schedule: The interleaving produces a result that no serial execution would produce. This is a bug.
Examples
EXAMPLE 1 — Good interleaving (serializable)

T1: A=A+100, [gap], B=B-100.
T2: [gap], A=1.06×A, [gap], B=1.06×B.
Order: T1:R(A), T1:W(A), T2:R(A), T2:W(A), T1:R(B), T1:W(B), T2:R(B), T2:W(B).
This is equivalent to running T1 then T2 serially. ✓ Serializable.

EXAMPLE 2 — Bad interleaving (not serializable)

T1: A=A+100, [gap], B=B-100.
T2: [gap], A=1.06×A, B=1.06×B.
Order: T1:W(A), T2:W(A), T2:W(B), T1:W(B).
T2 applies interest to A, then T1 overwrites it! T2's interest on A is lost. This result can't happen in any serial order. ✗ Not serializable.

EXAMPLE 3 — Why serializability is the goal

If every transaction is correct when run alone, then any serializable schedule is also correct — because it's equivalent to running them one at a time. The DBMS doesn't need to understand the business logic; it just ensures the schedule is serializable, and correctness follows automatically.

Three Concurrency Anomalies Slides 249–250

When transactions interleave incorrectly, three types of anomalies can occur based on the conflict type:

AnomalyConflictWhat HappensExample
Dirty Read WR (Write-Read) T2 reads data written by T1, but T1 aborts. T2 used data that never officially existed. T1 writes A=200, T2 reads A=200, T1 ABORTS (A goes back to 100). T2 made a decision based on a phantom value.
Unrepeatable Read RW (Read-Write) T1 reads A, T2 modifies A, T1 reads A again and gets a different value. T1 reads salary=5000. T2 gives raise, salary=6000, commits. T1 reads salary again: 6000! The same read returned two different values within one transaction.
Overwriting Uncommitted Data WW (Write-Write) T1 writes A, T2 overwrites A before T1 commits or aborts. T1 sets A=100 (not committed). T2 sets A=200 (commits). T1 aborts → rolls back A. But what value? A was 200 when T1 "rolled back" — T2's committed write is lost!
Examples
EXAMPLE 1 — Dirty read in real life

An airline reservation system: T1 tentatively books seat 12A (writes "reserved"). T2 reads the seat list, sees 12A is reserved, and shows the flight as full to another customer. T1 then aborts (payment failed). Seat 12A is actually free, but T2 gave wrong info. The customer might have bought a more expensive flight unnecessarily.

EXAMPLE 2 — Unrepeatable read

T1 is generating a report: reads total inventory = 1000 units. Meanwhile T2 sells 200 units and commits. T1 reads total inventory again (for verification): gets 800. The report has inconsistent numbers — 1000 in one place, 800 in another, for the "same" moment in time.

EXAMPLE 3 — WW conflict

T1 sets an employee's salary to $80,000. T2 (a bulk raise) sets it to $84,000 and commits. T1 aborts — rolls back salary. The rollback restores the "before T1" value, which might overwrite T2's committed $84,000 raise! T2's committed change is silently lost.

Strict Two-Phase Locking (Strict 2PL) Slide 251

Strict 2PL is the most common protocol for preventing all three anomalies. The rules:

  1. Before reading an object, acquire a S (shared) lock. Multiple transactions can hold S locks on the same object simultaneously — they're just reading, no conflict.
  2. Before writing an object, acquire an X (exclusive) lock. Only one transaction can hold an X lock. If any other transaction holds ANY lock (S or X), you must wait.
  3. All locks are held until the transaction commits or aborts — never released early. This is the "strict" part.

Strict 2PL guarantees that only serializable schedules are produced.

Examples
EXAMPLE 1 — S locks (readers don't block readers)

T1 reads account A (acquires S lock on A). T2 also wants to read A — no problem! S locks are compatible with S locks. Both read A simultaneously. But T3 wants to write A — T3 requests X lock on A, which is blocked until T1 and T2 both release their S locks (i.e., commit).

EXAMPLE 2 — X lock prevents dirty reads

T1 writes A (holds X lock on A). T2 wants to read A — requests S lock — blocked! T2 must wait until T1 commits or aborts. This prevents T2 from reading T1's uncommitted (possibly wrong) value. Dirty reads eliminated.

EXAMPLE 3 — Cascading aborts prevented

Without strict (holding locks until commit): T1 writes A, releases lock. T2 reads A (T1's uncommitted value), releases lock. T3 reads A (T1's uncommitted value). T1 aborts. Now T2 and T3 have used data that was rolled back — they might also need to abort. This "cascade" is expensive. Strict 2PL prevents it: no one can read T1's value until T1 commits.

The Log — How Aborts & Crashes Are Handled Slides 252–253

The log (also called the write-ahead log or WAL) is a sequential file that records every change made to the database. Before any change is written to the actual database, the log record is written first — hence "write-ahead."

Every log record for a write contains:

  • Transaction ID
  • Which object was changed
  • Before image: the old value (for undo)
  • After image: the new value (for redo)
📝
WAL Rule: Log must go to disk BEFORE the changed page

If the database page is written to disk but the log isn't, and a crash occurs — you can't undo the change. The log record must always be on disk first, so recovery can always undo uncommitted changes.

Examples
EXAMPLE 1 — Log for a bank transfer

T1 does A=A+100. Log entry: [T1, object=A, before=500, after=600]. Then T1 does B=B-100. Log entry: [T1, object=B, before=300, after=200]. Then T1 commits: [T1, COMMIT]. The log is a complete audit trail of every change.

EXAMPLE 2 — Using the log to abort

T1 partially runs (wrote A, hasn't written B yet), then user cancels. The DBMS reads T1's log entries backward: finds [T1, A, before=500, after=600] → restores A to 500. T1's effects are completely erased. The log records are chained by transaction ID so the DBMS can find all of T1's entries efficiently.

EXAMPLE 3 — Duplexed log

The log is often written to two different disk drives simultaneously (duplexed). If one disk fails, the other has the complete log. This ensures that even hardware failures don't prevent recovery. The log is the most critical file in a database system.

Crash Recovery — The ARIES Algorithm Slide 254

When the database crashes (power failure, OS crash, etc.) and restarts, it must restore a consistent state. The ARIES algorithm does this in three phases:

PhaseWhat It DoesDirection
1. Analysis Scan the log forward from the last checkpoint. Identify which transactions were active at crash time, and which database pages were "dirty" (changed but not yet written to disk). Forward →
2. Redo Redo ALL logged updates, even those from transactions that aborted. This ensures the database is in the exact state it was at the moment of crash (all changes replayed). Forward →
3. Undo Undo all changes by transactions that were still active (had not committed) at the time of the crash. Restores the "before" values from the log. Backward ←
Examples
EXAMPLE 1 — Why Redo before Undo?

At crash time, some committed changes might not have been flushed to disk yet (they were in the buffer pool). Redo ensures ALL committed changes are on disk. Then Undo removes uncommitted changes. If you did Undo first, you might undo changes that were from committed transactions not yet on disk.

EXAMPLE 2 — Analysis phase

Checkpoint was at time T=100. Crash at T=200. Analysis scans the log from T=100 to T=200. Finds: T1 committed at T=150 (safe), T2 was still active at T=200 (must undo), T3 wrote pages 5 and 7 (dirty pages, may need redo). After analysis, the recovery system knows exactly what work to do.

EXAMPLE 3 — Crash during recovery

What if the system crashes AGAIN while recovery is running? ARIES handles this gracefully. Because Redo replays all logged actions (idempotent), running Redo again doesn't cause double-application problems. And Undo records its own "compensation log records" (CLRs) so it knows where to resume if interrupted. Recovery is restart-safe.

🔒 Transaction Management Summary

  • Transaction: A sequence of reads/writes treated as one atomic unit.
  • Atomicity: All or nothing — no partial transactions visible.
  • Serializable schedule: Concurrent execution that produces the same result as some serial execution. The correctness criterion.
  • Three anomalies: Dirty Read (WR), Unrepeatable Read (RW), Lost Update (WW). All prevented by Strict 2PL.
  • Strict 2PL: S locks for reads, X locks for writes. Hold ALL locks until commit/abort. Guarantees serializability and avoids cascading aborts.
  • Write-Ahead Logging (WAL): Log record goes to disk before the changed page. Enables undo (abort) and redo (crash recovery).
  • ARIES: Analysis (find dirty pages and active transactions) → Redo (replay everything) → Undo (rollback uncommitted transactions). The standard crash recovery algorithm.
