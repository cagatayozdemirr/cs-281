What Does 3NF Actually Achieve? Slide ~158 ★ Slide Example

The slides dedicate a whole slide to this question that the first guide skipped. When 3NF is violated by an FD X→A, the slides prove that exactly one of two problems arises, and these problems reveal WHY 3NF is the right minimum standard.

Case 1: X is a proper subset of some key K

This means X is part of the key but not the whole key. The FD X→A says some attribute A depends on only part of the key. This causes redundant storage: the (X, A) pair gets repeated for every different combination of the rest of the key. For example, if key = {StudentID, CourseID} and Grade→InstructorName holds, every row for the same course repeats the instructor name redundantly.

Case 2: X is not a proper subset of any key

There's a FD chain: K → X → A. This means you can't store an (X, A) pair unless you also have a K value to associate with it. You can't record that X maps to A without committing to a full key K. This is the insertion anomaly problem — you can't store a fact about X independently of K.

⚠️
3NF is a compromise — problems can still arise even IN 3NF

The slides explicitly state: even if a relation IS in 3NF, these problems could still occur. 3NF just minimises them — it doesn't eliminate them entirely. That's why BCNF is always preferred if achievable.

Examples
★ Slide Example — Reserves SBDC (the canonical 3NF-but-not-BCNF case)

Relation: Reserves(S, B, D, C) = (SailorID, BoatID, Day, CreditCardID). FDs: S→C (each sailor has one credit card) and C→S (each card belongs to one sailor). Keys: {S,B,D} and {C,B,D} — two candidate keys.

The FD S→C violates BCNF (S is not a key). But C is part of key {C,B,D}, so 3NF is satisfied by Case 1 above. However: for every reservation by sailor S, the (S, C) pair is stored again and again. If a sailor makes 100 reservations, their credit card appears 100 times. This is 3NF redundancy that cannot be avoided without losing a dependency.

◆ Invented Example — Case 2: FD Chain

Relation: Enrollment(StudentID, CourseID, DeptID, Building). Suppose: CourseID→DeptID (each course belongs to one dept) and DeptID→Building (each dept is in one building). Key: {StudentID, CourseID}. Chain: {StudentID,CourseID} → CourseID → DeptID → Building.

You cannot store "the CS dept is in Engineering Hall" without also knowing of a specific enrollment (a StudentID+CourseID). That's Case 2 — the FD chain means facts about DeptID are trapped inside the Enrollment table.

◆ Invented Example — Why 3NF is still a useful compromise

In the Reserves(S,B,D,C) example, decomposing to BCNF would lose the dependency S→C or C→S (you'd need a join to check them). 3NF keeps the table intact, accepting some redundancy, but guaranteeing all FDs can be checked locally. In practice: if enforcement of all FDs matters more than storage efficiency, choose 3NF.

Refining an ER Diagram Using FD Analysis Slide ~169 ★ Slide Example

This topic connects everything together: you design an ER diagram, translate it to relations, then use FD/normalization analysis to discover that your ER diagram was wrong and improve it. Normalization feeds back into ER design.

💡
The key insight

Sometimes an FD you discover during normalization reveals that an attribute should have been placed in a DIFFERENT entity set in the ER diagram. Normalization is not just about splitting tables — it's a signal to reconsider the original conceptual design.

Examples
★ Slide Example — Workers and Departments (the exact slide example)

Before (initial ER translation):

Workers(S, N, L, D, since) -- ssn, name, lot, deptid, since Departments(D, M, B) -- deptid, managerssn, budget

The "lot" (L) is a parking lot assigned to a worker. It lives in Workers. Seems fine at first.

Discovered FD: All workers in the same department are assigned the same lot: D → L (department determines lot).

This is a BCNF violation in Workers (D is not a key for Workers). There's redundancy: every employee in Department 5 stores "Lot C" in their row. Fix by splitting:

Option A — Split Workers: Workers2(S, N, D, since) -- ssn, name, deptid, since Dept_Lots(D, L) -- deptid, lot Option B — Fine-tune (merge lot into Departments): Workers2(S, N, D, since) -- ssn, name, deptid, since Departments(D, M, B, L) -- deptid, manager, budget, lot

Option B is cleaner: "lot" belongs to Departments conceptually, which is where it should have been in the ER diagram from the start. Normalization revealed an ER design mistake.

◆ Invented Example — Phone number in wrong entity

Initial ER: Employees(EmpID, Name, DeptID, DeptPhone). Discovered FD: DeptID→DeptPhone (every employee in Dept 3 has the same phone 555-1234).

This reveals DeptPhone belongs to Departments, not Employees. Refine the ER diagram: remove DeptPhone from Employees entity set, add it to Departments. Then re-translate: Employees(EmpID, Name, DeptID) and Departments(DeptID, DeptName, DeptPhone). Now there's no redundancy and no update anomaly.

◆ Invented Example — Salary grade in wrong table

Initial: Employees(EmpID, Name, JobTitle, SalaryGrade, Salary). Discovered FD: JobTitle→SalaryGrade. This means salary grade should be in a Jobs table, not Employees. Refine: Employees(EmpID, Name, JobTitle) and Jobs(JobTitle, SalaryGrade, BaseSalary). The normalization result directly tells you to split the ER entity.

BCNF and Dependency Preservation — Sometimes Impossible Slide ~167 ★ Slide Example

The slides have a dedicated slide proving that a dependency-preserving decomposition into BCNF is not always possible. This is a critical theoretical result. When it happens, you must choose between BCNF (losing dependency preservation) and 3NF (keeping it, accepting some redundancy).

Examples
★ Slide Example — CSZ: the proof that BCNF can lose dependencies

Relation: CSZ (CityName, StreetName, ZipCode). FDs: CS→Z (city + street determines zip) and Z→C (zip determines city). Both candidate keys: {C,S} and {Z,S}.

Check BCNF violations: Z→C — is Z a superkey? No (Z alone doesn't determine S). So Z→C violates BCNF. Try to decompose: split into ZC (zip, city) and ZS (zip, street)? But then CS→Z cannot be checked on either ZC or ZS alone — you'd need a join. No BCNF decomposition preserves CS→Z.

This is the fundamental tension: BCNF vs dependency preservation. In this case, 3NF is the correct choice — CSZ is in 3NF (C is part of key {C,S}, so Z→C satisfies the 3NF condition "A is part of some key").

★ Slide Example — CSJDPQV: adding JPC just for constraint checking

From the slides: decomposition of CSJDQV into SDP, JS, and CJDQV is a lossless join but NOT dependency-preserving for FD JP→C (because J and P are now in different tables).

Solution: add table JPC to the collection of relations, purely so the FD JP→C can be checked locally. But this means JPC tuples are stored only for enforcement — pure overhead, pure redundancy. This is the real cost of BCNF when dependency preservation fails.

◆ Invented Example — Flight booking analogy

Imagine: FlightSeat(FlightNo, SeatNo, PassengerID). FDs: FlightNo+SeatNo → PassengerID (one passenger per seat per flight) and FlightNo+PassengerID → SeatNo (one seat per passenger per flight). Both form candidate keys.

Any BCNF decomposition splits the table. But to check the second FD (one seat per passenger), you'd need to join the pieces. No way around it. This is a CSZ-style situation. Accept 3NF — both FDs are checkable locally, with minimal redundancy.

3NF Decomposition Algorithm — Step by Step Slides ~170–172

The slides describe a concrete algorithm for decomposing into 3NF that always produces a lossless-join, dependency-preserving result. The key step is using a minimal cover instead of the original FD set.

Step 1: Find the Minimal Cover

A minimal cover G for a set of FDs F is the smallest set of FDs with the same closure as F. Every FD in G is needed, each has a single attribute on the right side, and no attribute on either side is redundant.

Properties of a minimal cover:

  • Closure of F = closure of G (same power).
  • Right-hand side of every FD in G is a single attribute.
  • If you remove any FD or any attribute from any FD in G, the closure changes — nothing is redundant.

Step 2: Create tables from minimal cover

For each FD X→A in the minimal cover, create a relation with attributes X ∪ {A}. If X→A and X→B are both in the cover, combine them into one table XAB with key X.

Step 3: Ensure lossless join

If no relation in the result contains a key for the original relation R, add a table containing just the key attributes of R.

🔑
Why minimal cover, not just F?

If you naively add table XY for every unpreserved FD, XY might itself violate 3NF. For example, if you add CJP to preserve JP→C, but you also have J→C, then within CJP the FD J→C violates 3NF (J is not a key of CJP). The minimal cover avoids this by stripping out redundant FDs first.

Examples
★ Slide Example — Computing a minimal cover

Given FDs: A→B, ABCD→E, EF→GH, ACDF→EG

Step 1: Make all RHS single attributes (decompose): A→B, ABCD→E, EF→G, EF→H, ACDF→E, ACDF→G

Step 2: Remove redundant left-side attributes. ABCD→E: since A→B, we know B is derivable from A. Can we drop B from ABCD? Check: does ACD→E hold? A→B, so ACD→B, and ABCD→E, so ACD→E. Yes! Remove B. ABCD→E becomes ACD→E. Continue: ACDF→E — does ACD→E already? Yes! Drop ACDF→E as redundant. Similarly ACDF→G — EF→G and ACD→E, so ACDF→E→G. Drop ACDF→G.

Step 3: Remove redundant FDs. Final minimal cover: A→B, ACD→E, EF→G, EF→H

This produces 3NF tables: AB (key A), ACDE (key ACD), EFG (key EF), EFH (key EF) — or combined: EFGH (key EF).

◆ Invented Example — Simple 3NF decomposition

Relation: SNLRWH (SSN, Name, Lot, Rating, Wage, Hours). FDs: S→SNLRWH, R→W. Minimal cover: {S→N, S→L, S→R, S→H, R→W} (simplify — S→W is derivable via S→R→W, so remove it from the cover).

3NF tables from minimal cover: one table per FD group. S determines N, L, R, H → SNLRH (key=S). R determines W → RW (key=R). Result: SNLRH and RW — exactly what we'd expect, and it's both lossless-join (shared column R) and dependency-preserving (each FD in its own table).

◆ Invented Example — When 3NF and BCNF decompositions differ

For CSZ: BCNF decomposition loses CS→Z. 3NF decomposition using minimal cover {CS→Z, Z→C}: creates tables CSZ (key={C,S} or {Z,S}) — which is already 3NF! No decomposition needed. The result preserves both FDs and is lossless. This shows why 3NF decomposition is preferred when BCNF loses dependencies: the 3NF algorithm simply keeps CSZ intact because it's already in 3NF.

Cost of Operations — Why Each Formula Is What It Is Slides ~180–183 ★ Slide Example

The original guide showed the cost table but didn't explain why each cell has the formula it does. Here is the reasoning for every entry. Recall: B = data pages, R = records per page, D = time per page I/O. Assumptions: indexes use 10% of record size for entries (Alt 2/3), hash has 80% occupancy → 1.25B pages, tree has 67% occupancy → 1.5B pages.

OrganizationOperationFormulaWhy this formula
(1) Heap Scan BD Read every page once sequentially. B pages × D cost each.
(1) Heap Equality 0.5BD On average, find the record halfway through a scan. Expected = B/2 pages. (Assumes exactly one match on a key.)
(1) Heap Range BD No ordering — must scan everything to find all matches.
(1) Heap Insert 2D Read last page (1D), write it back with new record (1D). Append to end of file.
(1) Heap Delete Search + D First find the record (equality search cost), then write the page back with record removed (1D).
(2) Sorted Scan BD Same as heap — read every page. Sorted order doesn't help scan cost.
(2) Sorted Equality D·log₂B Binary search over B pages. Binary search takes log₂B page comparisons.
(2) Sorted Range D·log₂B + #matches Binary search to find first match (log₂B), then scan forward page by page to read all matches.
(2) Sorted Insert Search + BD Find where to insert (log₂B), then shift all subsequent pages to maintain sort order — costs BD in worst case.
(3) Clustered B+ Scan 1.5BD All data is in the B+ tree (Alt 1). 67% occupancy means 1.5× more pages than the raw data.
(3) Clustered B+ Equality D·logF(1.5B) Traverse tree from root to leaf. Height ≈ logF(1.5B) where F is fanout (~100–200). Very fast.
(3) Clustered B+ Insert Search + D Find the right leaf (search cost), write the updated leaf page (1D). Occasional splits add overhead but amortized away.
(4) Unclustered Tree Scan BD(R + 0.15) BD to read all data pages. Plus 0.15B pages for the index itself (10% of data entry size × 1.5 for 67% occupancy). Each data record may be on a different page → terrible for scans.
(4) Unclustered Tree Equality D(1 + logF 0.15B) Traverse the index tree (logF of 0.15B index pages), then 1 fetch of the actual data page. Total: index height + 1.
(4) Unclustered Tree Range D·logF(0.15B) + #matches Find first match via tree, then each matching record may be on a DIFFERENT data page → one I/O per matching record. Very expensive for large result sets.
(5) Unclustered Hash Equality 2D 1D to hash and read the index bucket, 1D to fetch the actual data record. Constant time! Best equality performance.
(5) Unclustered Hash Range BD Hash scatters related keys to random buckets. Must scan ALL data pages — same as full heap scan. Hash is useless for range queries.
(5) Unclustered Hash Insert 4D Read index bucket (1D), write index bucket with new entry (1D), read data page (1D), write data page with new record (1D).
Examples
★ Slide Example — Specific file organizations compared

The slides set up the comparison using exactly these 5 organizations, all on the same relation with search key <age, sal>:

(1) Heap file (random order, insert at end-of-file) (2) Sorted file, sorted on <age, sal> (3) Clustered B+ tree, Alternative 1, search key <age, sal> (4) Heap file + unclustered B+ tree index on <age, sal> (5) Heap file + unclustered hash index on <age, sal>

The 5 operations compared are: (a) Scan all, (b) Equality search, (c) Range search, (d) Insert, (e) Delete. The "good enough to show trends" note is important — these are approximations, not exact measurements.

◆ Invented Example — Choosing between (3) and (4)

Your database has B=10,000 pages, R=100 records/page, D=10ms. A query does range search returning 500 matching records.

Clustered B+ (3): log133(15,000) ≈ 2.3, so about 3 I/Os to reach leaf + 500/100 = 5 pages of sequential data = ~8 I/Os = 80ms.

Unclustered tree (4): log133(1,500) ≈ 1.8 for tree + 500 random I/Os (one per record) = ~500 I/Os = 5,000ms.

Clustered is 62× faster for this range query. This is exactly why clustering matters so much.

◆ Invented Example — Why sorted insert costs Search + BD

Sorted file with B=1000 pages. Insert a new record that belongs on page 1 (the first page alphabetically). You must shift pages 2 through 1000 one position forward to make room — that's reading and writing 999 pages ≈ BD. Binary search to find the position costs D·log₂(1000)=10D. Total: ~2000D. Contrast with heap insert: just 2D. This is why sorted files are rarely used in practice — B+ trees give similar search performance with much cheaper inserts.

Understanding the Workload Before Choosing Indexes Slide ~184 ★ Slide Example

The slides say you must understand the workload before deciding which indexes to build. An index is useless if it doesn't serve the actual queries. This slide gives a checklist of what to analyse.

For each query in the workload, ask:

  • Which relations does it access?
  • Which attributes are retrieved (appear in SELECT)?
  • Which attributes are in selection/join conditions (appear in WHERE/ON)?
  • How selective are those conditions likely to be? (Does "age > 30" match 90% of rows, or 1%?)

For each update in the workload, ask:

  • Which attributes are in selection/join conditions?
  • How selective are those conditions?
  • What type of update (INSERT / DELETE / UPDATE)? Which attributes are affected?
✅
Why updates matter for index decisions

Every index you build must be maintained on every INSERT, DELETE, and UPDATE. A table with 5 indexes pays 5× the write cost on every modification. If a table is written 10,000 times per second but queried only 10 times, building extra indexes hurts overall throughput even if they speed up those 10 queries.

Examples
★ Slide Example — Clustered index choice for GROUP BY

Two queries from the slides on an Employees table:

Query 1: SELECT E.dno FROM Emp E WHERE E.age > 40 Query 2: SELECT E.dno, COUNT(*) FROM Emp E WHERE E.age > 10 GROUP BY E.dno

For Query 1: B+ tree on E.age seems useful. But how selective is age > 40? Depends on the data distribution. If most employees are over 40, even an index doesn't help much.

For Query 2: If many tuples have E.age > 10 (likely — almost everyone is over 10), using the E.age index retrieves most rows anyway. Then you still need to sort them by dno for the GROUP BY. A clustered index on E.dno is much better: GROUP BY dno is free (data already sorted by dno), and the WHERE age>10 is applied as a filter.

Lesson: Query 2 changes the optimal index choice completely. Understanding the GROUP BY changes the analysis. You must consider the whole query, not just the WHERE clause.

★ Slide Example — Equality queries with duplicates

Query from the slides:

SELECT E.dno FROM Emp E WHERE E.hobby = 'Stamps'

If many employees share the same hobby (hobby is not unique), a clustered index on E.hobby is extremely valuable. All employees with hobby='Stamps' are physically together on disk — one sequential scan of that group. An unclustered index would mean one random I/O per matching employee — terrible if 10,000 employees collect stamps.

◆ Invented Example — High-write table

A web server logs every HTTP request to a Requests table: 50,000 inserts per second. Analysts run daily reports filtering by user_id. You're tempted to add an index on user_id. But: 50,000 inserts/second means 50,000 index updates/second. The write overhead is enormous. Better solution: write to an unindexed log table, then periodically ETL to a reporting table that has the index. Understanding the write workload changes the architecture entirely.

Prefix Key Compression in B+ Trees Slide ~216 ★ Slide Example

Non-leaf pages in a B+ tree store keys only to direct searches — they don't store the actual data. This means we don't need the full key value stored in every non-leaf node. We just need enough information to decide which subtree to follow. Prefix key compression exploits this: store only a prefix of the key that is sufficient to separate adjacent subtrees.

Why does this matter? Higher fanout = fewer levels in the tree = fewer disk I/Os per search. Every byte saved in a non-leaf node allows more keys per page, increasing fanout F, reducing tree height logF(N).

⚠️
The critical correctness constraint from the slides

When compressing a key in a non-leaf node, the compressed key must still be greater than every key value in any subtree to its left. You can only compress as much as this constraint allows.

Examples
★ Slide Example — Dannon/David/Devarakonda

Three adjacent index entries in a non-leaf node: "Dannon Yogurt", "David Smith", "Devarakonda Murthy"

Can we compress "David Smith"? It needs to be greater than everything to its left (everything in the Dannon Yogurt subtree) and less than everything to its right (the Devarakonda Murthy subtree).

Naive guess: compress to "Dav" — is "Dav" > "Dannon Yogurt"? Yes. Is everything in the left subtree < "Dav"? Only if there's no key between "Dannon Yogurt" and "Dav". But what if there's a data entry "Davey Jones"? "Davey Jones" > "Dav", meaning "Dav" would incorrectly route a search for "Davey Jones" to the wrong subtree.

Correct answer: compress only to "Davi" — long enough to be strictly greater than "Davey Jones" (since "Davi" < "Davey"), so "Davey Jones" is still correctly routed. The rule: the compressed key must be the shortest prefix that is still strictly greater than all keys in the left subtree.

◆ Invented Example — Numeric key compression

Non-leaf node has entries: 100512, 200348, 300791. The actual minimum difference between them is large. Can compress 200348 to just "2" if the left subtree contains only numbers starting with 1 and the right subtree starts at 3. "2" correctly routes any key: keys starting with 1 go left, keys starting with 2 stay, keys starting with 3+ go right. 6 bytes compressed to 1 — massive fanout increase.

◆ Invented Example — Why fanout matters so much

Without compression: non-leaf keys are 50 bytes each, page size 4KB → ~80 keys per page → fanout ≈ 81. Height for 10M records: log₈₁(10,000,000) ≈ 3.8 → 4 levels.

With compression: keys compressed to ~10 bytes → ~400 keys per page → fanout ≈ 401. Height: log₄₀₁(10,000,000) ≈ 2.9 → 3 levels. One fewer disk I/O per search, guaranteed. For a database serving millions of queries per second, this is enormous.

A Note on 'Order' — Theory vs. Practice Slide ~220 ★ Slide Example

In theory, a B+ tree of order d means each node holds between d and 2d keys. The minimum occupancy rule is "at least d keys." In practice, real database systems don't use the order parameter. Instead they use a physical space criterion: "at least half-full."

Why the difference? Three reasons from the slides:

  • Index pages vs. leaf pages differ: A non-leaf page holding 4-byte integer keys fits many more entries than a leaf page holding full records. Order d would have to be different for different page types — impractical.
  • Variable-length records and keys: If records are different sizes, different nodes hold different numbers of entries. A fixed d doesn't make sense.
  • Duplicate keys with Alternative 3: Even with fixed-length keys, Alternative 3 stores a list of rids per key. If one key value has 1,000 duplicates, that entry is enormous. The same "order d" can't apply to all nodes.
Examples
★ Slide Example — Variable-length entries break fixed order

Index uses Alternative 3 (key + list of rids). Search key is a product name (variable-length string). Entry for "Apple" might have 50,000 matching records (50,000 rids). Entry for "ZZZ-Widget" might have 1 matching record. Both entries would be in the same B+ tree node, but one is thousands of times larger than the other. Saying "this node holds between d and 2d entries" is meaningless — the physical space used varies wildly per entry.

◆ Invented Example — How "half-full" works in practice

Page size: 8 KB. A non-leaf page stores 4-byte integer keys → could fit ~2000 keys per page. The "half-full" rule says: as long as this page uses at least 4 KB of its 8 KB capacity, it's fine. Whether that's 900 keys or 1100 keys depends on the actual key sizes of the specific keys in that node. No fixed d needed.

◆ Invented Example — Why this matters for exam problems

When exam questions say "B+ tree of order d=2," they mean each leaf/node holds between 2 and 4 entries. This is the textbook model used for small worked examples (like the insert/delete examples in the slides). In real databases, the order d concept is replaced by "at least half-full" — but for exam calculations with small trees, use d exactly as stated.

Linear Hashing as a Variant of Extendible Hashing

LH Described as a Variant of EH Slide ~233 ★ Slide Example

The slides have a dedicated slide making an important conceptual point: Linear Hashing (LH) and Extendible Hashing (EH) are mathematically the same scheme, just implemented differently. Understanding this connection shows why LH works without a directory.

The slide's argument, step by step:

  1. Start with an EH index whose directory has N elements.
  2. Instead of doubling the directory all at once when a bucket overflows, split buckets round-robin (one at a time, bucket 0 first, then 1, then 2…).
  3. When bucket 0 splits: in EH, you'd double the directory from N to 2N entries. But entries <1,N+1>, <2,N+2>, …, <N-1, 2N-1> all point to the same bucket as before — they're identical pairs. Only entry N is new (it points to the split image of bucket 0).
  4. So you only need to create directory element N, not double the whole directory.
  5. When bucket 1 splits next: create directory element N+1. And so on.
  6. The directory grows by one entry at a time instead of doubling. If you allocate primary bucket pages in order (bucket 0 at position 0, bucket 1 at position 1…), you can calculate the position of the i-th bucket directly — no directory pointer needed! You navigate to bucket i by calculating its position.
🔑
The key insight: LH eliminates the directory by exploiting sequential allocation

EH needs a directory because buckets can be scattered anywhere in memory. LH forces primary bucket pages to be allocated sequentially in order, so you can find bucket i directly by address. The "directory" is implicit in the sequential layout. This is why LH uses no directory structure but does need overflow pages — the overflow pages handle the cases that EH handles with split images in the directory.

Examples
★ Slide Example — The equivalence argument

Start with EH, N=4 buckets, directory = [00→B0, 01→B1, 10→B2, 11→B3].

Bucket 0 (pointed to by "00") overflows. EH would double to 8 entries: [000→B0, 001→B1, 010→B2, 011→B3, 100→B0_new, 101→B1, 110→B2, 111→B3].

But notice: entries 001=01, 010=10, 011=11 didn't change — they're identical to before, just with a leading 0. Only entry 100 is new. So instead of creating all 8 entries, just create entry "100" pointing to the split image. The other 7 are the same as before. This is exactly what LH does — grow one entry at a time, exploiting that most entries don't change.

◆ Invented Example — Sequential allocation as an implicit directory

Imagine a parking garage where cars are numbered sequentially: Car 0 is in Space 0, Car 1 in Space 1, etc. You don't need a map — you know exactly where Car 47 is. LH is like this: bucket i is always at disk address (base + i × page_size). No directory needed. EH is like a parking garage where cars park in random spots — you need a map (directory) to find Car 47.

◆ Invented Example — Trade-offs: when to choose LH vs EH

Choose EH when: Data distribution is skewed (many keys hash to same bucket) — EH concentrates splits on "hot" buckets. Directory fits in memory easily for uniform data. You want zero overflow pages.

Choose LH when: You want simpler implementation (no directory management). Data might have duplicates (LH handles well). Memory is very limited (no directory at all). You prefer gradual, predictable growth over sudden directory doublings.

Both are dynamic and both beat static hashing for growing data sets. In practice, most systems use extendible hashing variants because the directory doubling is fast and overflow pages are truly eliminated for non-duplicate keys.

The Exact T1/T2 Bank Transfer Schedules from the Slides Slides ~244–248 ★ Slide Example

The slides use a concrete bank transfer example throughout the transaction chapter. You need to know this exact example for the exam — the notation, the two schedules, and why one is OK and one is not.

The two transactions:

T1: BEGIN A = A + 100, B = B - 100 END (Transfer $100 from B's account to A's account) T2: BEGIN A = 1.06*A, B = 1.06*B END (Credit both accounts with 6% interest)

Both are submitted at the same time. The DBMS must produce a result equivalent to running them serially — either T1 then T2, or T2 then T1. That's the serializability requirement.

Schedule 1 — This is OK (serializable):

T1: A=A+100, B=B-100 T2: A=1.06*A, B=1.06*B Timeline →

T1 finishes updating A, then T2 reads the new A, then T1 updates B, then T2 reads the new B. This is equivalent to running T1 completely, then T2. Serializable ✓

Schedule 2 — This is NOT OK (not serializable):

T1: A=A+100, B=B-100 T2: A=1.06*A, B=1.06*B Timeline →

T2 reads and updates B before T1 applies the transfer to B. T1's update to B comes after T2 is done. T1's -100 overwrites T2's interest calculation on B. B loses its interest payment. This result cannot be produced by any serial ordering of T1 and T2. Not serializable ✗

The DBMS's abstract view of Schedule 2 (exactly as written in the slides):

T1: R(A), W(A), R(B), W(B) T2: R(A), W(A), R(B), W(B) The DBMS sees only reads R() and writes W() — not the arithmetic.
📐
R(X) and W(X) notation

The DBMS doesn't understand "A = A + 100." It only sees R(A) (read object A from disk to buffer) and W(A) (write buffer value of A back to disk). All the arithmetic happens in the application layer. The DBMS's job is to schedule R() and W() operations safely across transactions.

Serial schedules and equivalence — exact definitions from the slides:

  • Serial schedule: A schedule that does NOT interleave actions of different transactions. T1 runs completely, then T2 runs completely (or vice versa).
  • Equivalent schedules: For any database state, executing schedule 1 produces the identical database state as executing schedule 2. (Same final values for all objects.)
  • Serializable schedule: A schedule that is equivalent to SOME serial execution of the transactions. The schedule may interleave, but the end result must match a serial run.
Examples
★ Slide Example — The three anomalies shown as exact R/W schedules

The slides show all three anomalies using R/W notation. These are the exact layouts to memorise:

Anomaly 1 — Dirty Read (WR conflict): T1: R(A), W(A), R(B), W(B), Abort T2: R(A), W(A), Commit T2 reads A after T1 wrote it, but T1 later aborts. T2 used a value that was rolled back — a "dirty" value.
Anomaly 2 — Unrepeatable Read (RW conflict): T1: R(A), R(A), W(A), Commit T2: R(A), W(A), Commit T1 reads A, T2 changes A and commits, T1 reads A again. T1 gets TWO DIFFERENT VALUES for the same read within one transaction.
Anomaly 3 — Overwriting Uncommitted Data (WW conflict): T1: W(A), W(B), Commit T2: W(A), W(B), Commit T1 writes A (not committed). T2 also writes A and B and commits. T1 then commits, overwriting T2's committed W(B) with T1's value. T2's write to B is silently lost.
◆ Invented Example — Computing the result of Schedule 2

Start: A=1000, B=2000. Schedule 2 (bad interleaving):

1. T1: R(A), W(A) → A becomes 1100. 2. T2: R(A) reads 1100, W(A) → A becomes 1100×1.06=1166. 3. T2: R(B) reads 2000, W(B) → B becomes 2000×1.06=2120. T2 Commits. 4. T1: R(B) reads 2120, W(B) → B becomes 2120-100=2020. T1 Commits.

Final: A=1166, B=2020. Total=3186.

But correct serial order T1→T2 gives: A=1100×1.06=1166, B=1900×1.06=2014. Total=3180. Different! The interleaving "created" 6 extra dollars — money appears from nowhere. This is why non-serializable schedules are wrong.

◆ Invented Example — Verifying Schedule 1 is serializable

Start: A=1000, B=2000. Schedule 1 (OK interleaving):

1. T1: R(A), W(A) → A=1100. 2. T2: R(A) reads 1100, W(A) → A=1166. 3. T1: R(B), W(B) → B=1900. 4. T2: R(B) reads 1900, W(B) → B=2014.

Final: A=1166, B=2014. Total=3180. This matches T1 then T2 serially: A: 1000+100=1100→×1.06=1166 ✓. B: 2000-100=1900→×1.06=2014 ✓. Serializable confirmed.

Aborting a Transaction — Cascading Aborts and Why Strict 2PL Prevents Them Slide ~251 ★ Slide Example

The slides dedicate a full slide to this topic that the original guide folded into the locking discussion. It deserves separate treatment because it explains the "strict" in "Strict 2PL."

When transaction Ti is aborted, all its changes must be undone — that's atomicity. But there's a more dangerous problem: if Tj has already read a value that Ti wrote (and Ti is now aborting), Tj must ALSO be aborted. Otherwise Tj made decisions based on data that was rolled back — it's built on a lie. This is called a cascading abort.

⚠️
Cascading aborts are catastrophic

Imagine Tj read Ti's dirty value and already committed. Now Ti aborts. You can't un-commit Tj — it already told the user "success!" You have a permanently inconsistent database. This is why the "strict" rule exists: prevent this possibility entirely.

The solution: Strict 2PL holds ALL locks until commit time.

If Ti holds an X lock on object A until it commits (or aborts), then Tj cannot read A until Ti commits. If Ti aborts, Tj never saw Ti's dirty write — no cascading abort is possible.

The slides also explain how this connects to crash recovery: the DBMS maintains a log of all writes. At crash time, all transactions that were active (not yet committed) are aborted using the log. Since Strict 2PL prevents cascading aborts, you only need to undo the active transactions — not a potentially infinite chain of dependent transactions.

Examples
★ Slide Example — The cascading abort scenario

From the slides exactly:

T_i: writes object A ... later aborts T_j: reads object A (T_i's uncommitted value) ... must also abort! Without Strict 2PL: T_j can read A immediately after T_i writes it. With Strict 2PL: T_i holds X lock on A until commit/abort. T_j must wait. If T_i aborts, T_j never reads A. Cascading abort prevented.

The slide states: "If Ti writes an object, Tj can read this only after Ti commits." This is the core guarantee of the strict policy.

★ Slide Example — Crash recovery connection

From the slides: "This mechanism [the log] is also used to recover from system crashes: all active transactions at the time of the crash are aborted when the system comes back up."

This connects Strict 2PL to ARIES recovery: because Strict 2PL prevents cascading aborts, when the system recovers from a crash, the UNDO phase only needs to undo transactions that were active at crash time. There's no need to hunt for a potentially infinite chain of transactions that read dirty values from aborted transactions. The system's correctness guarantees are bounded and tractable.

◆ Invented Example — The cascade without Strict 2PL

Without strict locking: T1 writes A=500 (not committed). T2 reads A=500 and makes a decision based on it, writes B=1000 (commits). T3 reads B=1000, commits. Now T1 aborts — A rolls back to A=300. T2's decision was wrong (it used A=500, not A=300) → T2 must abort. But T2 already committed! And T3 read T2's B=1000 which came from T2's wrong decision → T3 might also need aborting. The cascade can propagate through an arbitrary number of committed transactions. Catastrophic.

◆ Invented Example — Why "strict" adds to "2PL"

Regular 2PL: acquire all locks before releasing any (two phases: growing, shrinking). This prevents non-serializable schedules. But it allows releasing locks before commit — a transaction can release A's lock as soon as it's done with A, even before committing. Another transaction can then read A. If the first transaction later aborts → cascading abort risk.

Strict 2PL: hold ALL locks until commit/abort. No lock is ever released early. This adds the no-cascading-abort guarantee on top of serializability. The cost: slightly lower concurrency (other transactions must wait longer). The benefit: clean, safe abort behaviour and simple crash recovery.

🔒 Transaction Management — Complete Picture

  • Transactions = sequences of R() and W() operations. The DBMS doesn't understand application semantics — only reads and writes.
  • Serializability = the correct-concurrency criterion. A schedule is correct if it produces the same result as some serial execution.
  • Three anomalies: Dirty Read (WR), Unrepeatable Read (RW), Lost Update (WW). Each has a specific R/W schedule pattern shown in the slides — memorise them.
  • Strict 2PL: S locks before reads, X locks before writes. All locks held until commit/abort. Prevents all three anomalies AND cascading aborts.
  • Cascading aborts: If Ti aborts and Tj read Ti's dirty value, Tj must also abort. Strict 2PL prevents this entirely by never allowing reads of uncommitted data.
  • The log: Records (old value, new value) for every write. Log must go to disk BEFORE the changed page (WAL). Enables undo of aborted transactions and crash recovery.
  • ARIES crash recovery: Analysis → Redo (replay everything) → Undo (rollback active transactions). Strict 2PL makes Undo bounded — only active transactions need undoing.
